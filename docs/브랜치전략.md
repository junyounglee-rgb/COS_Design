# 브랜치 전략

## 1. 브랜치 구조

| 브랜치 | 용도 |
|--------|------|
| `main` | 개발 브랜치 (xlsx 관리) |
| `release_v01` | 라이브 브랜치 |
| `release_v01_hotfix` | 핫픽스 브랜치 |
| `release_v02 ~ v04+` | 다음 패치 준비 브랜치 |

---

## 2. xlsx 구조 변경

### target 컬럼 추가

| target 값 | 의미 |
|-----------|------|
| `all` | 모든 브랜치에 적용 (기본값) |
| `v03` | v03만 적용 |
| `v03,v04` | v03, v04만 적용 |
| `v05+` | v05 이후 전부 적용 |

> 기본값은 `all`, 예외인 행만 명시

### 같은 id가 all과 ver 둘 다 있을 경우

```
ver 데이터가 우선 적용 (all 데이터 덮어씀)
```

예시:

| id | name | hp | target |
|----|------|----|--------|
| 001 | 전사 | 1000 | all |
| 001 | 전사 | 1200 | v02 ← v02에서는 이 데이터 사용 |
| 002 | 마법사 | 800 | all |

## 3. 로컬 폴더 구조

```
main 브랜치
├─ char.xlsx
├─ monster.xlsx
└─ output/
    ├─ all/           ← 공통 데이터 (all 태그)
    │   ├─ char.pb
    │   └─ char.csv
    ├─ v02/           ← v02 전용 데이터만 (차이 있을 때만 생성)
    │   ├─ char.pb
    │   └─ char.csv
    └─ v03/           ← v03 전용 데이터만
        ├─ char.pb
        └─ char.csv
```

### 파일 우선순위 규칙

```
ver 폴더 파일 있음 → ver 파일 사용
ver 폴더 파일 없음 → all 파일 사용
```

> 차이가 있는 버전만 폴더 생성, 나머지는 all로 대체

---

## 4. CSV를 사용하는 이유

### .pb 파일의 한계

```
.pb 파일은 바이너리 파일
└─ git이 내용을 줄 단위로 비교 불가
└─ diff 확인 불가
└─ cherry-pick 불가
└─ 잘못 들어간 데이터를 눈으로 확인할 방법 없음
```

### .csv가 필요한 이유

| 항목 | .pb | .csv |
|------|-----|------|
| git diff | ❌ 불가 | ✅ 가능 |
| 사람이 읽기 | ❌ 불가 | ✅ 가능 |
| 변경 내용 검증 | ❌ 불가 | ✅ 가능 |
| 서비스에서 사용 | ✅ 사용 | ❌ 미사용 |

### diff 예시

```
Fork 툴에서 output/v03/char.csv diff 확인

- 전사, 1000, 0.05, fire   ← 이전 데이터
+ 전사, 1200, 0.03, fire   ← 변경된 데이터
+ 마법사, 800, 0.05, ice   ← 새로 추가된 데이터
```

> **.pb는 실제 서비스용, .csv는 검증용** 으로 역할이 명확히 분리됨
> datasheet 실행 시 항상 두 파일을 동시에 생성

## 5. datasheet 수정 사항 (개발자 요청)

> `datasheet` : xlsx → .pb 변환 내부 프로그램

- `--branch` 인자 받기
- target 컬럼 필터링 로직 추가
- `.pb` + `.csv` 동시 생성
- 로컬에서도 실행 가능하게 구성

### target 필터링 규칙

```
all       → 무조건 포함
v05+      → 버전 번호가 05 이상이면 포함
v05       → v05 정확히 일치할 때만 포함
v03,v04   → 목록에 포함될 때만 포함
```

## 6. 작업 프로세스

### 6-1. 사전 준비 (최초 1회)

```
1. xlsx에 target 컬럼 추가
   └─ 기존 행 전부 target: all 로 채우기

2. 커밋 컨벤션 팀 공유

3. datasheet 수정 요청 (개발자에게)
   └─ target 필터링 + .pb + .csv 동시 생성
   └─ 로컬 실행 가능하게

4. GitHub Actions 설정 (개발자에게)
   └─ xlsx 변경 감지 시 자동 실행
   └─ output/ 각 브랜치에 자동 배포
```

### 6-2. 일반 데이터 작업 (매번)

```
1. main 브랜치 최신화 (fetch & pull)

2. 작업 브랜치 생성 (신규 큰 피쳐 개발 시)
   └─ 브랜치명 예시: data/monster-boss-stats
   └─ 단순 데이터 업데이트는 main 직접 작업 가능

3. xlsx 수정
   └─ 데이터 추가/수정
   └─ target 컬럼 확인

4. 로컬에서 datasheet 실행
   └─ output/ 폴더에 전체 생성

5. Fork 툴 Local Changes에서 CSV diff 확인
   └─ output/all/char.csv
   └─ output/v03/char.csv   ← 브랜치 이동 없이 한 화면에서 확인
   └─ output/v04/char.csv

6. 이상 없으면 커밋 (커밋 컨벤션 준수)

7. GitHub에 Push & PR

8. GitHub Actions 자동 배포
   └─ output/all/  → 모든 릴리즈 브랜치
   └─ output/v03/  → release_v03 브랜치
   └─ output/v04/  → release_v04 브랜치
```

### 6-3. 핫픽스 작업

```
1. release_v01_hotfix 브랜치에서 작업 브랜치 생성

2. xlsx 수정
   └─ target: v01 로 지정

3. 커밋 후 PR → release_v01_hotfix 로 머지

4. 동일 내용 main에도 PR 올려서 반영
```

---

## 7. PR 올리기 전 체크리스트

```
□ target 컬럼 올바르게 지정했는가
□ 로컬 datasheet 실행했는가
□ CSV diff 확인했는가 (의도한 변경만 있는지)
□ 커밋 메시지 양식 맞는가
□ Version 태그 있는가
□ Wrike 태그 있는가
□ main 최신화 후 작업했는가
□ 커밋은 목적별로 분리했는가
```

---

## 8. 핵심 원칙

| 원칙 | 내용 |
|------|------|
| xlsx는 main에서만 관리 | 브랜치별 xlsx 편집 금지 |
| .pb는 datasheet로만 생성 | 직접 수정 금지 |
| .csv는 diff 확인용 | 서비스 미사용 |
| PR 없이 main 직접 푸시 금지 | 반드시 리뷰 후 머지 |
| 커밋은 목적별 분리 | 여러 파일 한번에 커밋 금지 |
| all은 기본값 | ver 행은 예외값 |
