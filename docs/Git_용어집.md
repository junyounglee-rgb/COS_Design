# Git 용어집
> 이 문서는 Git을 처음 접하는 분들을 위한 용어 설명입니다.

---

## Git이란?

> 쉽게 말해 **"파일의 변경 역사를 기록하고 여러 사람이 함께 작업할 수 있게 해주는 도구"** 입니다.

```
비유: 구글 문서의 "버전 기록" 기능과 비슷하지만
      훨씬 강력하고 여러 사람이 동시에 작업 가능
```

---

## Git을 왜 쓰는가?

### 핵심 이유 3가지

**1. 실수해도 되돌릴 수 있다**
```
비유: 작업 내용이 전부 "자동 저장 + 되감기" 가 되는 것
      잘못된 데이터를 배포했을 때 이전 시점으로 즉시 복구 가능
      커밋 단위로 정확하게 롤백 가능
```

**2. 누가 언제 왜 바꿨는지 추적할 수 있다**
```
비유: CCTV처럼 모든 변경 이력이 기록됨
      "이 데이터 누가 바꿨어?" → git log로 즉시 확인
      "언제부터 이 버그가 생겼지?" → 커밋 히스토리로 추적

SVN도 이력 관리는 되지만
Git은 커밋 메시지, 작업자, 날짜, Wrike 일감까지 연결 가능
```

**3. 여러 사람이 동시에 작업할 수 있다**
```
비유: 구글 문서처럼 여러 사람이 동시에 편집
      단, 각자 브랜치에서 독립적으로 작업 후 합치는 방식

SVN: 내가 파일 열면 다른 사람은 수정 불가 (잠금 방식)
Git: 각자 브랜치에서 동시에 작업 후 나중에 합침 (병렬 방식)
```

---

## SVN vs Git, 무엇이 다른가?

### 가장 큰 차이: 중앙 집중 vs 분산

```
SVN (중앙 집중)
┌─────────────────────────────┐
│         중앙 서버            │
│  (여기 없으면 아무것도 못함)  │
└──────────────┬──────────────┘
               │
    ┌──────────┼──────────┐
    ▼          ▼          ▼
  사람A       사람B       사람C
(서버 연결 필수)

Git (분산)
  사람A ←──→ 사람B ←──→ 사람C
    ↑                     ↑
    └──────── GitHub ──────┘
              (공유용)

→ 각자 PC에 전체 히스토리 보유
→ 서버 없어도 커밋, 브랜치 생성 가능
→ GitHub는 공유를 위한 공간일 뿐
```

---

### 브랜치 개념의 차이

```
SVN 브랜치
= 폴더를 통째로 복사하는 것
= 무겁고 시간이 오래 걸림
= 한번 만들면 오래 유지하는 경향

예시:
/trunk          ← 메인
/branches/v01   ← v01 폴더 전체 복사
/branches/v02   ← v02 폴더 전체 복사
/branches/v03   ← v03 폴더 전체 복사 ... 계속 쌓임

Git 브랜치
= 특정 커밋을 가리키는 "포인터" (이름표)
= 매우 가볍고 즉시 생성/삭제 가능
= 작업 후 머지하고 삭제하는 것이 정상

예시:
main ← 포인터
feature/boss ← 포인터 (작업 후 삭제)
hotfix/dragon ← 포인터 (수정 후 삭제)
```

---

### 머지(합치기) 능력의 차이

```
SVN 머지
= 선형적 구조 (순서대로 쌓임)
= 충돌이 적지만 동시 작업이 어려움
= 바이너리 파일도 상대적으로 무난

Git 머지
= 비선형적 구조 (여러 갈래가 합쳐짐)
= 강력한 자동 머지 (텍스트 파일 기준)
= 바이너리 파일은 머지 불가 (pb, xlsx)
  → csv를 중간 단계로 쓰는 이유
```

---

### 우리 회사 현재 상황

```
현재: Git을 쓰고 있지만 SVN 방식으로 사용 중

release_helsinki2   ← SVN의 /branches/v02 와 동일한 사고방식
release_helsinki3   ← SVN의 /branches/v03 와 동일한 사고방식

문제점:
1. 브랜치가 버전마다 계속 쌓임
2. 커밋 메시지가 부실해 히스토리 추적 불가
3. pb 파일이 바이너리라 Git의 머지 기능 활용 불가
4. PR/리뷰 없이 직접 push → 실수 검증 불가

개선 방향:
1. 브랜치는 작업용, 버전은 태그로 관리
2. 커밋 컨벤션 도입 (Wrike, Version 태그)
3. xlsx → csv → pb 구조로 머지 가능하게
4. PR + CI/CD로 자동 검증 및 배포
```

## 1. 기본 개념

### Repository (저장소) - "프로젝트 보관함"
> 프로젝트의 모든 파일과 변경 역사가 저장되는 공간

```
비유: 팀 전체가 공유하는 구글 드라이브 폴더
      단, 누가 언제 뭘 바꿨는지 모든 기록이 남음

예시: GitHub의 cos-data 저장소
      = 데이터 파일 전체 + 지금까지의 모든 수정 기록
```

---

### Branch (브랜치) - "나만의 작업 공간"
> 원본을 건드리지 않고 독립적으로 작업할 수 있는 공간

```
비유: 중요한 문서를 복사해서 내 자리에서 수정하는 것
      수정이 완료되면 원본에 반영, 실패하면 그냥 버림

예시:
main                    ← 공식 원본 (함부로 건드리면 안 됨)
feature/boss-monster    ← 보스 몬스터 작업용 내 공간
hotfix/dragon-hp        ← 드래곤 hp 버그 수정용 공간
```

---

### Commit (커밋) - "저장 + 메모"
> 변경 내용을 저장소에 기록하는 행위. 메모(커밋 메시지)를 함께 남김

```
비유: 작업 일지에 "오늘 뭘 했는지" 기록하는 것
      나중에 이 기록만 봐도 무슨 작업인지 알 수 있어야 함

좋은 예시: data(char): 전사 hp 1000 → 1200 수정 (Version: v02, Wrike: #1234)
나쁜 예시: [홍길동] 수정함
```

---

### Push (푸시) - "GitHub에 올리기"
> 내 PC에서 작업한 커밋을 GitHub 서버에 올리는 행위

```
비유: 내 자리에서 작업한 파일을 공유 드라이브에 올리는 것
      Push 전까지는 내 PC에만 있고 팀원은 볼 수 없음

순서: 내 PC에서 작업 → 커밋 → Push → GitHub에 반영
```

---

### Pull (풀) - "최신 내용 받아오기"
> GitHub 서버의 최신 내용을 내 PC로 가져오는 행위

```
비유: 공유 드라이브에서 최신 파일을 내 자리로 다운로드
      작업 시작 전에 항상 Pull 먼저 해야 함
      안 하면 내가 옛날 버전을 수정하는 상황 발생

습관: 작업 시작 전 → 반드시 Pull 먼저!
```

---

### Fetch (페치) - "확인만 하기"
> 서버에 변경사항이 있는지 확인만 하고 실제로 가져오지는 않음

```
비유: 공유 드라이브에 새 파일이 올라왔는지 "확인"만 하는 것
      다운로드(Pull)는 아직 안 함

Pull과 차이:
Fetch = "새 파일이 있네" (확인만)
Pull  = "새 파일 있네, 내 자리로 가져와야지" (실제 반영)
```

---

### Clone (클론) - "처음 받아오기"
> 저장소를 처음으로 내 PC에 복사하는 행위

```
비유: 입사 첫날 공유 드라이브 전체를 내 PC에 처음 다운로드
      이후에는 Pull로 변경사항만 받아오면 됨

언제: 새 PC 세팅할 때, 처음 프로젝트 참여할 때
```

---

## 2. 협업 개념

### PR (Pull Request) - "합쳐도 될까요? 검토 요청"
> 내 브랜치 작업을 main에 합쳐달라고 팀원에게 요청하는 것

```
비유: 내가 수정한 문서를 팀장에게
      "이거 공식 문서에 반영해도 될까요?" 라고 올리는 것
      팀장(리뷰어)이 검토 후 OK하면 반영(머지)됨

순서:
1. 내 브랜치에서 작업 완료
2. GitHub에서 PR 올리기 (버튼 클릭)
3. 팀원이 내용 검토
4. 승인 → main에 합쳐짐
```

---

### Merge (머지) - "합치기"
> 두 브랜치의 내용을 하나로 합치는 것

```
비유: 두 사람이 각자 수정한 문서를 하나로 합치는 것
      서로 다른 부분을 수정했으면 자동으로 합쳐짐
      같은 부분을 수정했으면 Conflict(충돌) 발생

예시: feature/boss-monster 작업 완료
      → PR → 승인 → main에 머지
      → feature/boss-monster 브랜치 삭제
```

---

### Conflict (충돌) - "같은 곳을 동시에 수정했을 때"
> 두 브랜치에서 같은 파일의 같은 부분을 수정했을 때 발생

```
비유: 두 사람이 같은 문서의 같은 문장을 동시에 다르게 수정
      → 어느 쪽을 쓸지 사람이 직접 결정해야 함

예시:
A가 char.csv 1행 전사 hp: 1000 → 1200 수정
B가 char.csv 1행 전사 hp: 1000 → 1500 수정
→ 둘 다 머지하려 하면 충돌 발생
→ 1200과 1500 중 어느 값을 쓸지 사람이 결정

예방법: 같은 파일을 동시에 작업하지 않도록 팀원과 협의
```

---

### Review (리뷰) - "내용 검토"
> PR로 올라온 변경 내용을 팀원이 확인하고 의견을 남기는 것

```
비유: 팀장이 보고서를 검토하고 코멘트 남기는 것

예시: "이 캐릭터 데이터 target이 v03인데 v04도 포함해야 하지 않나요?"
      "hp 값이 기획서랑 다른 것 같습니다"
```

---

### Approve (승인) - "OK, 합쳐도 됩니다"
> 리뷰 후 문제없으면 머지를 허락하는 것

```
비유: 팀장이 보고서 검토 후
      "이거 공식 반영해도 됩니다" 도장 찍는 것

승인 후: main에 머지 가능 상태가 됨
```

---

## 3. 버전 관리 개념

### Tag (태그) - "이 시점에 출시했음"
> 특정 커밋 시점에 이름표를 붙이는 것. 주로 출시 버전 표시에 사용

```
비유: 중요한 문서에 "2024년 3월 1일 최종본" 스티커를 붙이는 것
      나중에 그 시점으로 언제든 돌아갈 수 있음

브랜치와 차이:
브랜치 = 계속 작업하는 공간 (내용이 계속 바뀜)
태그   = 특정 시점의 스냅샷 (내용이 바뀌지 않음)

예시: helsinki3 출시 시점에 tag: helsinki3 생성
      → 나중에 "helsinki3 출시 당시 데이터가 뭐였지?" 바로 확인 가능
```

---

### diff (디프) - "뭐가 바뀌었는지 비교"
> 두 파일 또는 두 브랜치 간의 변경 내용을 비교하는 것

```
비유: 두 문서를 나란히 놓고 달라진 부분에 형광펜 칠하는 것

예시:
- 전사, 1000, 0.05   ← 이전 (빨간색)
+ 전사, 1200, 0.03   ← 변경 (초록색)

바이너리 파일(.pb, .xlsx)은 diff 불가
텍스트 파일(.csv)은 diff 가능 → csv를 쓰는 이유
```

---

### cherry-pick (체리픽) - "이 커밋만 골라서 가져오기"
> 특정 커밋 하나만 골라서 다른 브랜치에 적용하는 것

```
비유: 케이크에서 체리만 골라 먹는 것

예시: main에서 작업한 내용 중
      "드래곤 스탯 수정" 커밋 하나만 release/helsinki3에 적용

주의: .pb처럼 바이너리 파일은 cherry-pick 불가
      → csv 기반으로 관리해야 cherry-pick 활용 가능
```

---

### Rollback (롤백) - "이전 상태로 되돌리기"
> 잘못된 변경을 이전 커밋 상태로 되돌리는 것

```
비유: 문서 편집 중 Ctrl+Z로 실행 취소하는 것
      단, git은 훨씬 오래된 시점으로도 돌아갈 수 있음

예시: 드래곤 hp가 0으로 잘못 배포됨
      → 이전 커밋으로 롤백 → 정상 데이터 복구

커밋을 자주, 잘게 남겨야 롤백이 정확해짐
```

---

### Hotfix (핫픽스) - "긴급 수정"
> 라이브 서버에서 발생한 긴급 버그를 빠르게 수정하는 것

```
비유: 방송 중 자막 오타 발견 → 즉시 수정 후 재송출

일반 작업과 차이:
일반 작업: main → feature 브랜치 → 검토 → 머지 (시간 여유 있음)
핫픽스:    release 브랜치 → hotfix 브랜치 → 즉시 수정 → 배포 (긴급)

예시: 운영 중 드래곤 hp가 0으로 잘못 들어간 경우
      → hotfix/dragon-hp 브랜치 생성
      → 긴급 수정 후 release/helsinki3에 반영
      → 동일 내용 main에도 반영 (누락 방지)
```

## 4. 자동화 개념

### CI (Continuous Integration) - "자동 검증"
> Push 또는 PR 시 자동으로 실행되는 검사 과정

```
비유: 원고 제출 시 자동으로 맞춤법 검사, 분량 확인이 실행되는 것
      통과해야만 편집자(리뷰어)에게 넘어감

우리 프로젝트 예시:
xlsx 수정 후 Push
→ CI 자동 실행
→ "xlsx, csv, pb 내용이 일치하는가?" 자동 확인
→ ❌ 불일치 → 머지 차단 (사람이 놓친 실수를 자동으로 잡아줌)
→ ✅ 일치   → 머지 허용
```

---

### CD (Continuous Deployment) - "자동 배포"
> CI 검증 통과 후 자동으로 배포하는 과정

```
비유: 검수 통과한 상품이 자동으로 포장되어 각 매장으로 배송되는 것

우리 프로젝트 예시:
CI 통과
→ CD 자동 실행
→ output/all/  → 모든 release 브랜치에 자동 배포
→ output/v03/  → release_v03 브랜치에 자동 배포
```

---

### CI/CD - "검증부터 배포까지 전 과정 자동화"

```
전체 흐름:
xlsx 수정 → Push → PR
                    │
                    ▼
              CI 자동 검증
              (xlsx↔csv↔pb 일치 확인)
                    │
                ✅ 통과
                    │
                    ▼
              리뷰 & 승인
                    │
                    ▼
              CD 자동 배포
              (각 브랜치에 pb 자동 반영)
```

---

### GitHub Actions - "CI/CD 실행 도구"
> GitHub에서 제공하는 자동화 도구. CI/CD를 설정하는 데 사용

```
비유: 자동화 레시피를 등록해두면 조건에 맞을 때 자동 실행되는 시스템

예시 레시피:
"xlsx 파일이 변경되면"
→ datasheet 실행해서 csv, pb 생성
→ 각 브랜치에 자동 배포
```

## 5. 파일 관련 개념

### Binary (바이너리) - "기계만 읽을 수 있는 파일"
> 사람이 텍스트 편집기로 열어도 읽을 수 없는 파일 형식

```
예시: .pb, .xlsx, .png, .exe
특징: git으로 diff, merge, cherry-pick 불가
      → 변경 내용을 눈으로 확인할 방법 없음
      → 충돌 발생 시 해결 불가
```

---

### Plain Text (텍스트) - "사람도 읽을 수 있는 파일"
> 메모장으로 열어도 내용을 읽을 수 있는 파일 형식

```
예시: .csv, .json, .txt, .md
특징: git으로 diff, merge, cherry-pick 가능
      → 변경 내용을 줄 단위로 확인 가능
      → csv를 중간 단계로 쓰는 이유가 바로 이것
```

---

### Single Source of Truth - "단 하나의 기준 원본"
> 모든 데이터의 기준이 되는 단 하나의 원본

```
비유: 회사 공식 직인이 찍힌 계약서 원본
      복사본이 여러 개 있어도 이것만이 진짜

우리 프로젝트:
xlsx → csv → pb 구조에서
csv가 Single Source of Truth
pb는 csv로부터 자동 생성되는 산출물
```

---

### 산출물 (Artifact) - "자동으로 만들어지는 결과물"
> 원본 파일로부터 도구가 자동으로 생성하는 파일

```
비유: 워드 파일(원본)을 PDF로 변환한 것(산출물)
      PDF를 직접 수정하지 않고 항상 워드에서 수정 후 재변환

우리 프로젝트:
csv (원본) → datasheet 실행 → pb (산출물)
pb는 직접 수정하지 않음
항상 csv 수정 후 datasheet로 재생성
```

## 6. SVN vs Git 한눈에 비교

> 우리 회사는 SVN에서 Git으로 전환했지만 아직 SVN 방식으로 사용 중

| 항목 | SVN (이전 방식) | Git (올바른 활용) |
|------|----------------|-----------------|
| 브랜치 | 폴더 복사본처럼 무겁고 오래 유지 | 가볍고 작업 후 바로 삭제 |
| 버전 보관 | 브랜치로 버전별 보관 | 태그로 스냅샷, 브랜치는 작업용 |
| 머지 | 선형적, 충돌 적음 | 비선형, 강력한 자동 머지 |
| 협업 | 중앙 서버에 의존 | PR 기반, 리뷰 후 머지 |
| 바이너리 | 선형 구조라 상대적으로 괜찮음 | 머지 불가 → csv 중간 단계 필요 |

## 7. 우리 팀 작업 흐름 한눈에 보기

```
1.  Pull            → main 최신화 (작업 전 필수)
2.  브랜치 생성     → 내 작업 공간 만들기 (큰 피쳐일 때)
3.  xlsx 수정       → 데이터 작업
4.  datasheet 실행  → csv + pb 자동 생성
5.  diff 확인       → Fork 툴에서 csv 변경 내용 검토
6.  Commit          → 변경 내용 저장 (커밋 컨벤션 준수)
7.  Push            → GitHub에 올리기
8.  PR              → 팀원에게 검토 요청
9.  CI 자동 검증    → xlsx↔csv↔pb 일치 확인
10. Approve         → 팀원 승인
11. Merge           → main에 반영
12. CD 자동 배포    → 각 브랜치에 pb 자동 배포
```
