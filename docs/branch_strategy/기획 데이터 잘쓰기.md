## 2. 바이너리 파일의 근본 한계 — SVN처럼 쓸 수밖에 없는 이유

```
코드 파일 (.cs, .go 등):
  A브랜치: foo.cs 10번 줄 수정
  B브랜치: foo.cs 20번 줄 수정
  → git merge: 두 수정을 자동으로 합침 ✅

기획 데이터:
  A브랜치: cookies.xlsx 수정 (바이너리)
  B브랜치: cookies.xlsx 다른 쿠키 수정 (바이너리)
  → git merge: "어느 파일 전체를 쓸 것인가?" 양자택일 ❌
               라인 단위 merge 없음, cherry-pick 없음
```

**xlsx와 pb 모두 바이너리 파일이다.**
이 때문에 발생하는 필연적인 문제:

- 같은 파일을 두 사람이 동시에 수정하면 한 사람 작업이 통째로 사라짐
- 브랜치 간 특정 행(특정 쿠키 데이터)만 골라서 이동하는 것이 불가능
- `git blame`으로 "이 수치를 누가 언제 바꿨는지" 추적 불가

### 이미 시도한 것들 — textconv와 xlmerge

팀에서 바이너리 한계를 극복하기 위해 이미 도구를 만들어 적용했다.

```
.gitattributes:
  *.xlsx diff=xlsx merge=xlsx

.gitconfig (로컬):
  diff.xlsx.textconv = manager/manager textconv -excel
  merge.xlsx.driver  = manager/xlmerge %O %A %B %A %P
```

**textconv (텍스트 diff)** — Fork에서 xlsx 파일의 변경 내역을 텍스트로 볼 수 있다.
`manager/manager textconv -excel` 명령이 xlsx를 텍스트로 변환하여 git diff에 표시한다.
이 덕분에 Fork에서 xlsx 파일의 "무엇이 바뀌었는지"를 확인할 수 있다.

**xlmerge (엑셀 병합)** — xlsx 파일의 3-way merge를 시도하는 커스텀 merge 드라이버다.
`update xlmerge` 커밋이 여러 번 존재하며, 지속적으로 개선해온 흔적이 있다.

**하지만 한계가 있다:**
- xlmerge는 수식(`=SUM`, `=VLOOKUP`)이 포함된 셀에서 깨지는 경우가 보고됨
- textconv는 diff 표시용일 뿐, merge 자체를 해결하지는 못함
- 3개월간 `Merge branch` 커밋 34건 → xlmerge가 있어도 충돌이 빈번하게 발생

### 왜 SVN처럼 쓸 수밖에 없는가 — 인과 관계

바이너리 파일의 특성이 연쇄적으로 현재 구조를 강제했다.

```
xlsx, pb = 바이너리
    │
    ├─► git merge/cherry-pick 불가
    │       → 버전별로 파일을 통째로 분리해야 함
    │       → 브랜치 = SVN release branch (파일 복사본)
    │
    └─► xlsx와 pb가 항상 쌍으로 동기화되어야 게임이 작동
            → 누군가 pb를 깨뜨리면 이후 모든 작업이 정합성 문제 발생
            → CI에서 xlsx:pb 동기화 검증을 강제로 차단
```

### 실제 발생한 CI 차단 사례

**Case 1: xlsx↔pb 비동기화**
```
[release_helsinki3] 데이터 업로드에 실패했습니다 @Junyoung So

업로드한 엑셀과 pb 파일의 싱크가 맞지 않습니다:
  protobuf/ui_strings.pb
Error: xlsx:pb not synced
```
→ 한 명이 datasheet.exe 미실행 상태로 push → 해당 브랜치 전체 팀 배포 차단

**Case 2: 데이터 유효성 오류**
```
[main] 데이터 업로드에 실패했습니다 @heejun.choi

Validation error: quests.wasm:
  quests:
    quest 31: goal_type: no fields are set in oneof
    quest 32: goal_type: no fields are set in oneof
    ...
```
→ 퀘스트 데이터 필드 누락 → pb는 생성됐지만 내용이 잘못됨 → 전체 차단

**Case 3: datasheet --build 옵션 미존재 확인**
```
flag provided but not defined: -target
Usage of datasheet-linux-amd64:
  -cos-common-path string
  -cos-data-path string
  -validate
```
→ 누군가 `-target` 옵션을 시도했으나 datasheet에 해당 기능이 없음
→ **`datasheet --build` 옵션이 현재 존재하지 않는다는 실증**. 개발 요청이 반드시 필요하다.

### CI 차단 구조의 딜레마

CI가 강제 차단하는 건 맞는 선택이다. pb가 깨진 채로 배포되면 게임이 터진다.
그런데 이 차단이 **브랜치를 공유하는 모든 기획자의 push를 동시에 막는다.**

```
기획자 A가 pb 비동기 상태로 push
    │
    ▼
CI 차단 (xlsx:pb not synced)
    │
    ▼
기획자 B, C, D ... 도 같은 브랜치에서 push 불가
    │
    ▼
A가 수정해서 다시 push할 때까지 전체 대기
```

바이너리 파일 구조 + 단일 브랜치 공유 → CI 차단이 팀 전체에 전파되는 구조.
이것이 Revert 61건이 발생한 배경이다.

## 3. 현재 구조

```
기획자 A ──► excel/cookies.xlsx (release_helsinki3 브랜치)
                     │ 수정 후
                     ▼
              datasheet.exe 실행
                     │
                     ▼
             protobuf/cookies.pb 생성
                     │
                     ▼
              commit + push

기획자 A ──► excel/cookies.xlsx (release_helsinki2 브랜치)  ← 같은 작업 반복
기획자 A ──► excel/cookies.xlsx (release_helsinki_vng 브랜치) ← 또 반복
기획자 A ──► excel/cookies.xlsx (main 브랜치) ← 또 반복
```

**같은 수정을 브랜치 수만큼 반복해야 한다.**

브랜치가 늘어날수록 전파해야 할 브랜치도 늘어난다.
해외 퍼블리셔가 추가되면 5개, 6개가 될 수도 있다.

### 라이브 배포 후 — H2 브랜치의 역할 변화

H3 개발 중 H2가 라이브로 나가면, `release_helsinki2`는 **핫픽스 전용 브랜치**로 전환된다.

```
라이브 전:  release_helsinki2 = H2 기능 개발 브랜치
라이브 후:  release_helsinki2 = H2 핫픽스 베이스 (신규 기능 없음)
```

이 운영 방식 자체는 맞다 — 검증된 라이브 베이스 위에서 핫픽스가 이루어지므로 안전하다.

**그러나 바이너리 한계는 핫픽스에도 똑같이 적용된다.**

H2에서 핫픽스를 적용하면, H3과 main에도 같은 수정이 필요하다.
코드 저장소라면 `cherry-pick` 한 줄로 끝나지만, cos-data는 수동 재작업이다.

```
H2 핫픽스:  release_helsinki2에서 수정 → 커밋
               ↓ cherry-pick 불가 (바이너리)
H3 적용:    release_helsinki3 체크아웃 → 동일 작업 수동 반복
main 적용:  main 체크아웃 → 동일 작업 수동 반복
```

Phase 3 완성 이후에는 main 하나에서 핫픽스하면 CI가 모든 버전 pb를 자동 재빌드한다.
핫픽스용 별도 브랜치가 필요 없어진다.


### 수동 전파의 실체 — cherry-pick도 안 된다

"그냥 cherry-pick하면 되지 않나?"라고 생각할 수 있다. 실제로는 안 된다.

`cos-client` 실측 사례:

```
ca3f4697  [h3] 대기실 캐릭터 세팅 크기 조정 [Wrike:4428729103]  ← h3 브랜치
c1f88f5f  [h3] 대기실 캐릭터 세팅 크기 조정 [Wrike:4428729103]  ← main 브랜치
```

같은 작업자, 같은 커밋 메시지, 같은 Wrike 번호. 그런데 **tree 해시가 다르다.**

```
ca3f4697 tree: 4c2238e8...
c1f88f5f tree: 2c05be11...   ← cherry-pick이면 같아야 함
```

왜 다른가 — 두 브랜치의 파일 시작 상태가 이미 달랐기 때문이다:

```
h3 브랜치 기준 수정 전:          main 브랜치 기준 수정 전:
  _positionOffset: {x: 0}         _positionOffset: {x: -0.1, y: 0.3}  ← 다름
  _scaleOffset: 1.27              _scaleOffset: 1.2                    ← 다름
  _scaleOffset: 1.19              _scaleOffset: 1.2                    ← 다름

목표값 (둘 다 동일):
  _scaleOffset: 1.17
```

cherry-pick을 시도했다면 conflict가 났을 것이다.
브랜치 간 파일 역사가 달라서, diff를 그대로 다른 브랜치에 적용할 수 없다.

**결론: 작업자가 h3에서 작업한 뒤, main에서 같은 의도의 작업을 손으로 다시 했다.**
cherry-pick도 아니고, 파일 복사도 아닌 — **수동 재작업**이 "수동 전파"의 실체다.

## 5. $filter 컬럼 — 이미 설계되어 있다

`keywords.xlsx`의 build 시트에 버전별 우선순위가 이미 정의되어 있다.

| 키 | 값 | 적용 범위 |
|----|----|-----------|
| `$$LAUNCH_0` | 1 | 베이스 — 모든 버전 공통 |
| `$$HELSINKI_3` | 1030 | H3 이상에서 override |
| `$$ISTANBUL_1` | 2010 | IST1 이상에서 override |
| `$$ISTANBUL_3` | 2030 | IST3 이상 |
| `$$ISTANBUL_4` | 2040 | IST4 이상 |
| `$$JEJU_1` ~ `$$JEJU_3` | 3010~3030 | JEJU 버전대 |
| `$$KYOTO_1` ~ `$$KYOTO_2` | 4010~4020 | KYOTO 버전대 |
| `$$VNG_ONLY` | 9999999999 | VNG 전용 |
| `$$DEV` | 9999999999 | 개발 전용 ⚠️ VNG_ONLY와 값 동일 |
| `$$NONE` | 10000000000 | 비활성 (어떤 빌드에도 포함 안 됨) |

> JEJU, KYOTO까지 이미 설계되어 있으므로 $filter 전략은 장기 확장 가능하다.

> ⚠️ **DEV와 VNG_ONLY가 같은 우선순위 값(9999999999)이다.**
> `--build DEV` 실행 시 VNG_ONLY 행도 포함되고, 반대도 마찬가지다.
> `_oneTarget` 접미사로 해결하거나, 값을 분리하는 협의가 필요하다.

**규칙: 숫자가 높은 $filter가 낮은 $filter를 덮어쓴다.**

### 사용 예시 — 까망베르 쿠키 HP

| id | hp | $filter | 설명 |
|----|-----|---------|------|
| 1000 | 100 | `$$LAUNCH_0` | 베이스: 모든 버전에서 HP=100 |
| 1000 | 120 | `$$HELSINKI_3` | H3 이상에서는 HP=120으로 override |
| 1000 | 150 | `$$ISTANBUL_1` | IST1 이상에서는 HP=150으로 override |

- H2 빌드 결과: HP = 100
- H3 빌드 결과: HP = 120
- IST1 빌드 결과: HP = 150

기획자는 **main 하나의 파일 안에서 모든 버전의 수치를 관리**한다.

### 현재 $filter 설계의 한계 — "이상" 개념만 있다

현재 $filter는 **최솟값(floor)** 개념이다.

```
$$HELSINKI_3 = 1030
  → H3 빌드(1030) 이상이면 전부 포함
  → Istanbul(2010), VNG도 함께 포함됨
```

**H3에만 들어가고 Istanbul에는 빠져야 하는 데이터**를 표현할 수 없다.

| 필요한 경우 | 예시 | 현재 가능 여부 |
|------------|------|---------------|
| H3 이상 전부 적용 | 신규 쿠키 추가 | ✅ `$$HELSINKI_3` |
| H3 전용 (Istanbul 제외) | H3 한정 이벤트, H3 임시 밸런스 | ❌ 표현 불가 |
| Istanbul 이상 | 다음 버전 신규 기능 | ✅ `$$ISTANBUL_1` |
| VNG 전용 | 지역 서버 전용 데이터 | ✅ `$$VNG_ONLY` |

### 제안: `_oneTarget` 접미사로 단일 버전 한정

컬럼 추가나 스키마 변경 없이, 기존 `$filter` 값의 **네이밍 컨벤션**으로 해결한다.

```
$$HELSINKI_3            → H3 이상 전부 포함 (기존 동작 유지)
$$HELSINKI_3_oneTarget  → H3 빌드에만 포함, Istanbul 이상 제외
```

**datasheet 동작 규칙:**

| $filter 값 | 판정 방식 | H3 빌드 | IST1 빌드 |
|-----------|----------|---------|----------|
| `$$HELSINKI_3` | 빌드 값 ≥ 필터 값 | ✅ | ✅ |
| `$$HELSINKI_3_oneTarget` | 빌드 값 == 필터 값 | ✅ | ❌ |
| `$$ISTANBUL_1` | 빌드 값 ≥ 필터 값 | ❌ | ✅ |

**장점:**
- 컬럼 추가 없음 — 기존 xlsx 구조 그대로 유지
- keywords.xlsx build 시트에 `$$HELSINKI_3_oneTarget` 항목만 추가
- 기존 데이터에 영향 없음 (하위 호환)

> 이 설계는 개발팀과의 협의가 필요하다. 섹션 10 요청서 참고.

## 6. 전환 로드맵

### Phase 1: 현재 구조 안정화 (즉시)

> 개발팀 도움 없이 즉시 적용 가능한 항목들

**6-1. 커밋 메시지 컨벤션 + 팀 공지**

현재 Wrike 번호 포함 커밋이 cos-data 기준 0.4%(17건/4,558건)다.
cos-client는 8.5%(1,270건/14,914건)로 개발팀은 어느 정도 습관화되어 있으나,
기획 데이터 저장소는 거의 사용되지 않고 있다.
`integrate-wrike.yml`이 이미 구현되어 있지만, 커밋 메시지에 번호가 없으니 작동하지 않는다.

```
현재: [김율희] 코인사용량 퀘스트 반영
목표: [기획/김율희][H3] 코인사용량 퀘스트 반영 [Wrike:4428xxx]
```

- **버전 태그**: `[H3]`, `[H2]`, `[VNG]`, `[ALL]`, `[HF]`(핫픽스), `[Test]`(테스트 데이터)
- **이름 prefix 제거 권장**: git이 Author를 자동 기록하므로 중복
- `.gitmessage` 템플릿 배포 + `git config commit.template .gitmessage` 적용

**`[Test]` 태그가 필요한 이유**

3개월 실측에서 아래와 같은 커밋들이 다수 발견됐다:

```
[김려강] test 랜덤팩
[김려강] 테스트
[김려강] 출석패키지test
[김률희] 글로벌 데일리미션 테스트 데이터
[이동성] 광장 쿨타임 90% 감소 테스트 데이터
[기획/오대송] 모드스케줄 테스트
[이동성] 쿨타임 시트 함수 임시 제거
```

이 커밋들은 나중에 원복 예정이거나, 검증 후 삭제 예정인 임시 데이터다.
태그 없이 커밋하면 나중에 `git log --grep`으로 필터링할 때 실제 작업 로그와 섞여서 구분이 불가능하다.

```
# 개선 후
[Test][H3] 광장 쿨타임 90% 감소 테스트 데이터
[Test][H3] 모드스케줄 테스트
[Test] 출석패키지 랜덤팩 검증용

# 나중에 테스트 커밋만 필터링
git log --grep="\[Test\]"

# 실제 작업 로그만 보기 (테스트 제외)
git log --grep="\[H3\]\|\[H2\]\|\[ALL\]\|\[HF\]"
```

**테스트 커밋 원칙**: 테스트 데이터를 push했다면 반드시 원복 커밋도 `[Test]` 태그와 함께 남긴다.
원복 예정이라면 push 전에 로컬에서 처리하는 것이 가장 좋다.

**6-2. 레거시 브랜치 정리**

운영이 끝난 브랜치 7개가 잔존한다: `release-essen`, `release-fargo`, `release-geneva`, `release_dubai`, `release_helsinki`, `test-main` 등.

- 절차: `git tag archive/<브랜치명>` → 팀 컨펌 → `git push origin :<브랜치명>`
- 효과: 잘못된 브랜치 push 사고 예방, 브랜치 목록 가독성 향상

### Phase 2: 파이프라인 전환 (개발팀 협의 후 최대한 빠르게)

> **개발팀 개발 요청 필요**
> Phase 2는 단일 브랜치 전환의 핵심이다. 여기가 완성되어야 Phase 3이 가능하다.
> 예상 기간: `datasheet --build` 구현 + CI 연동 완료까지 (개발팀 일정에 따름)

#### 현재 파이프라인

```
기획자 로컬:
  xlsx 수정 → datasheet.exe 수동 실행 → pb 생성 → xlsx + pb 함께 커밋
                                          ↑ 이걸 빠뜨리면 CI 차단
```

- pb를 로컬에서 수동 생성 → 사람이 빠뜨리는 구간 존재
- pb를 git에 직접 저장 → 저장소 용량 증가, 바이너리 충돌

#### 목표 파이프라인

```
기획자 로컬:
  xlsx 수정 → commit + push (xlsx만)
  → 끝. datasheet.exe 로컬 실행 없음. pb 커밋 없음.

CI가 자동으로:
  xlsx + $filter 기반으로 버전별 pb 생성
    → datasheet --build HELSINKI_3 → h3.pb → H3 Portal 배포
    → datasheet --build ISTANBUL_1 → ist1.pb → IST1 Portal 배포
    → datasheet --build VNG_ONLY   → vng.pb → VNG S3 배포
```

#### 개발팀 요청 사항

**요청 1: datasheet --build 옵션**

| 항목 | 내용 |
|------|------|
| 기능 | `$filter` 컬럼 기반 행 필터링 |
| 입력 옵션 | `--build <키>` (예: `--build HELSINKI_3`) |
| 필터 규칙 | `$filter` 값 ≤ 빌드 키 값인 행만 포함 (높은 값 override) |
| `_oneTarget` 접미사 | `==` 비교로 해당 빌드에만 포함 (섹션 5 참고) |
| `$filter` 없는 행 | 모든 빌드에 포함 (하위 호환) |
| `--build` 미사용 시 | 기존 동작 그대로 유지 (하위 호환) |
| 파일럿 대상 | `$filter` 컬럼이 실제 사용 중인 파일 (예: `cookies.xlsx`, `modes.xlsx`) |

#### 주의: cos-common 의존성

datasheet는 `-cos-common-path` 옵션으로 cos-common 저장소를 참조한다.
`--build` 옵션 구현 시 cos-common 브랜치 매칭 로직이 영향받는지 확인 필요.
현재 CI(upload.yaml)에서도 cos-common을 체크아웃하고 있다.

#### Phase 2 완료 시 변화

| 항목 | 현재 | Phase 2 완료 후 |
|------|------|----------------|
| 기획자 로컬 작업 | xlsx 수정 → datasheet.exe → xlsx+pb 커밋 | xlsx 수정 → xlsx만 커밋 |
| pb 생성 | 기획자 로컬에서 수동 | CI가 자동 |
| pb 비동기화 CI 차단 | 매번 발생 가능 | 구조적으로 불가능 (CI만 pb 생성) |
| 버전별 pb 분기 | 브랜치별 독립 관리 | `--build` 옵션으로 CI가 자동 분기 |

### Phase 2.5: xlsx → CSV 파이프라인 (검토 보류)

> **라이브 서비스 특성상 실익이 낮아 우선순위에서 제외.**
> Phase 3 완전 전환 이후, 팀 필요에 따라 별도 논의.

#### 왜 보류하는가

CSV 파이프라인은 pb와 동일하게 머티리얼라이즈드 per-version 구조다.

```
xlsx
  → h3/cookies.csv   →  h3/cookies.pb
  → ist1/cookies.csv →  ist1/cookies.pb
  → vng/cookies.csv  →  vng/cookies.pb
```

CSV는 "사람이 읽을 수 있는 pb"로, 1:1 대응이 성립한다.
base 데이터(`$$LAUNCH_0`) 수정 시 모든 버전 CSV가 함께 바뀌는 것은 **노이즈가 아니라 사실**이다 — 실제로 모든 버전 pb가 바뀌기 때문이다.

**보류 이유는 현재 우선순위와 개발 공수의 문제다.**

- Phase 2(`--build` 옵션)만 완성해도 기획자 일상 고통의 핵심이 해결된다
- CSV 파이프라인은 추가 개발 공수가 크고 (변환기 + datasheet CSV 입력 지원 + CI 수정), Phase 2 안정화 이후에 추진하는 것이 순서에 맞다
- textconv로 Fork diff가 이미 가능한 현재 상황에서 Phase 2 전에 투자할 필요가 없다

#### 도입 조건

Phase 3(단일 브랜치 완전 전환) 안정화 이후, 아래 수요가 실감될 때 재검토한다.

- "이 수치 누가 언제 바꿨는지" blame 수요가 높아질 때
- 동시 작업 merge 품질 개선이 필요할 때

### Phase 3: main 단일 브랜치 완전 전환

Phase 2 완성 이후 진행. 파일럿 2주 안정 확인 후 전체 파일 순차 전환.

#### 핫픽스 워크플로우 변화

Phase 3 이후 `release_*` 브랜치는 더 이상 기획자가 편집하지 않는다.
핫픽스 운영 방식도 바뀐다.

| | 현재 | Phase 3 이후 |
|--|------|-------------|
| 핫픽스 대상 브랜치 | `release_helsinki2` 체크아웃 후 수정 | `main`에서 수정 |
| H3/main 전파 | 수동 재작업 (브랜치마다 반복) | CI가 `--build HELSINKI_2` pb 자동 재빌드 |
| 핫픽스 브랜치 필요 여부 | 필요 (라이브 버전 격리) | 불필요 (main 하나에서 모든 버전 통제) |

#### 전환 전략 — 별도 브랜치 검증 후 일괄 전환

파일 단위 점진적 전환 없이 **별도 브랜치에서 전체 검증 후 일괄 적용**한다.
공존 기간 없이 한 번에 전환하므로 validate_diff.mjs 예외 처리 등 과도기적 복잡성이 없다.

#### $filter 없는 파일 처리

`$filter` 컬럼이 없는 xlsx는 모든 행을 `$$LAUNCH_0` (ALL)로 간주한다.
별도 마이그레이션 없이 기존 파일 그대로 사용 가능 — 전환 부담이 거의 없다.

```
$filter 컬럼 있는 파일  → 버전별 행 분리 그대로 동작
$filter 컬럼 없는 파일  → 전체 행이 모든 빌드에 포함 (기존 동작과 동일)
```

#### 롤백 계획

```
롤백 트리거: 검증 브랜치에서 pb 생성 오류 또는 게임 데이터 불일치 발견

롤백 절차:
  1. CI에서 --build 단계 비활성화 (upload.yaml 수정)
  2. 기존 방식(로컬 datasheet)으로 pb 재생성 후 커밋
  소요 시간: 30분 이내

안전장치:
  - --build 옵션 없이 기존 동작 보존 (하위 호환 설계)
  - 검증 브랜치에서 충분히 확인 후 main 적용
```

#### $filter 방식 전환 리스크 및 해결 방안

**리스크 1: $filter 값 누락 또는 오입력**

기획자가 잘못된 $filter 값을 입력하면 의도치 않은 버전에 데이터가 포함된다.

> 현재 구조도 완벽한 보호막이 아니다. 수동 전파 실수로 인한 Revert가 4개월간 61건 발생했다.
> 실수의 종류가 "전파 누락"에서 "$filter 오입력"으로 바뀔 뿐이며, CI로 사전 차단이 가능하다.

| 해결 방안 | 내용 | 개발 공수 |
|----------|------|----------|
| **$filter 유효성 검증 (CI)** | `keywords.xlsx` build 시트에 없는 $filter 값이면 CI 즉시 차단 | 낮음 |
| **버전별 pb validation (CI)** | `--build` 별로 생성된 pb를 각각 validate — 현재보다 검증 강화 | 낮음 (기존 로직 재사용) |

**리스크 2: "이 행이 어느 버전에 실제로 들어갔는가" 확인 어려움**

현재는 브랜치 CI 통과 = 해당 버전 데이터 맞음이라는 직관적 보장이 있다.
$filter 방식에서는 pb를 직접 열어보지 않으면 포함 여부가 불명확하다.

| 해결 방안 | 내용 | 개발 공수 |
|----------|------|----------|
| **datasheet --build --preview** | pb 생성 없이 "H3 빌드에 포함되는 행 목록"을 텍스트 출력 — 커밋 전 기획자 확인용 | 중간 |
| **PR 시 버전별 변경사항 자동 표시** | CI가 PR에 "H3 기준 변경 행", "IST1 기준 변경 행" 자동 코멘트 | 높음 |

> `--preview`만 구현해도 리스크 2는 대부분 해소된다.

```
main / release_helsinki2 / release_helsinki3 / release_helsinki_vng
```

xlsx와 pb 파일이 모두 바이너리이기 때문에 git merge/cherry-pick이 불가능하다.
결과적으로 기획자가 같은 데이터를 **브랜치 수만큼 반복 작업**해야 하며,
전파 실수로 인한 Revert가 4개월간 61건 발생했다.

향후 해외 퍼블리셔가 추가될수록 브랜치는 5개, 6개로 늘어나고,
수동 전파 부담도 선형으로 증가한다. 근본적인 구조 개선이 필요하다.

### 요청 기능: `datasheet --build <TARGET>` 옵션

#### 현재 동작
```
datasheet.exe
  → 모든 xlsx 읽기
  → 버전 구분 없이 모든 행 포함
  → pb 생성
```

#### 요청하는 동작
```
datasheet.exe --build HELSINKI_3
  → xlsx 읽기
  → $filter 컬럼 값 ≤ HELSINKI_3 빌드 우선순위(1030)인 행만 포함
  → pb 생성
```

#### $filter 컬럼 규칙 (keywords.xlsx build 시트에 이미 정의됨)

| 키 | 우선순위 값 | 포함 조건 (--build HELSINKI_3 기준) | 판정 방식 |
|----|------------|--------------------------------------|----------|
| `$$LAUNCH_0` | 1 | ✅ 포함 (1 ≤ 1030) | `≥` 비교 |
| `$$HELSINKI_3` | 1030 | ✅ 포함 (1030 ≤ 1030) | `≥` 비교 |
| `$$HELSINKI_3_oneTarget` | 1030 | ✅ 포함 (1030 == 1030) | `==` 비교 |
| `$$ISTANBUL_1` | 2010 | ❌ 제외 (2010 > 1030) | `≥` 비교 |
| `$$ISTANBUL_1_oneTarget` | 2010 | ❌ 제외 (2010 ≠ 1030) | `==` 비교 |
| `$$VNG_ONLY` | 9999999999 | ❌ 제외 | `≥` 비교 |

**기본 규칙**: `$filter` 우선순위 값 ≤ 빌드 타겟 우선순위 값인 행만 출력

**`_oneTarget` 규칙**: 접미사가 붙으면 `==` 비교 — 해당 빌드에만 포함, 상위 버전에서 제외

#### 입력 스펙

| 항목 | 내용 |
|------|------|
| 옵션명 | `--build <키>` |
| 키 값 예시 | `HELSINKI_3`, `ISTANBUL_1`, `VNG_ONLY`, `LAUNCH_0` |
| 키 소스 | `keywords.xlsx` build 시트 (기존 파일 그대로 참조) |
| `$filter` 컬럼 없는 행 처리 | 모든 빌드에 포함 (하위 호환) |
| `--build` 옵션 없을 때 | 기존 동작 그대로 유지 (하위 호환) |

#### 출력 스펙

| 항목 | 내용 |
|------|------|
| pb 포맷 | **기존과 동일** (형태 변경 없음) |
| 출력 경로 | 기존 `protobuf/*.pb` 경로 유지 또는 별도 지정 가능하게 |

### 파일럿 제안

전체 129개 xlsx를 한 번에 전환하지 않고, **`$filter` 컬럼이 실제 사용 중인 파일 1개로 파일럿** 진행을 제안한다.

- 파일럿 기준: `$filter` 컬럼이 있고, 버전별 데이터 분기가 실제로 발생하는 파일 (예: `cookies.xlsx`, `modes.xlsx`)
- `ui_strings.xlsx`는 스트링 특성상 `$filter` 없이 전 버전 동일 배포 — 별도 개선 과제로 분리
- 파일럿 성공 후 `product.xlsx` 등으로 순차 확대

### 기대 효과

| 효과 | 현재 → 목표 |
|------|------------|
| 기획자 작업 브랜치 수 | 3~4개 → **1개 (main)** |
| 버전별 데이터 전파 작업 | 브랜치마다 수동 복사 → **CI 자동 처리** |
| Revert 발생 원인 중 "전파 실수" 제거 | 61건/4개월 → 대폭 감소 |
| 해외 퍼블리셔 추가 시 부담 | 브랜치 추가마다 선형 증가 → **추가 부담 없음** |

### 우선순위 및 제약

- **우선순위**: 단일 브랜치 전환의 핵심 블로커 — 이 기능 없이는 구조 개선 불가
- **제약**: pb 파일 포맷 변경 없음, 기존 `--build` 미사용 시 동작 보존 필수
- **협의 필요**: 출력 경로 구조 (`protobuf/*.pb` 유지 vs `output/<버전>/*.pb` 분리)

### 추가 설계 요청 — `_oneTarget` 접미사로 단일 버전 한정

현재 `$filter` 단일 컬럼으로는 **특정 버전에만 한정**되는 데이터를 표현할 수 없다.

```
필요한 케이스:
  H3 한정 이벤트 → H3 빌드에 포함, Istanbul 빌드에서 제외
  H3 임시 밸런스 → H3에서만 적용, 이후 버전에는 다른 값 적용
```

**컬럼 추가 없이** `$filter` 값의 네이밍 컨벤션으로 해결하는 방식을 제안한다.

```
$$HELSINKI_3            → 기존 동작: H3 이상 전부 포함
$$HELSINKI_3_oneTarget  → 신규: H3 빌드에만 포함, Istanbul 이상 제외
```

**datasheet 구현 요청:**
- `$filter` 값이 `_oneTarget`으로 끝나면 `==` 비교 (정확히 일치하는 빌드만)
- `_oneTarget` 없으면 기존 `>=` 비교 유지 (하위 호환)
- `keywords.xlsx` build 시트에 `$$HELSINKI_3_oneTarget` 항목 추가만 필요 (스키마 변경 없음)

## 11. 더 나아가 — git답게 쓰기

> 이 섹션은 기획 데이터 단일 브랜치 전환(Phase 3)이 완료된 이후의 이야기다.
> 현재 구조가 왜 SVN처럼 운영되는지는 섹션 1.3, 2 참고.

### Phase 3 이후 git답게 쓸 수 있는 것들

**main 단일 브랜치 + $filter + datasheet --build** 가 완성되면
release 브랜치는 더 이상 "기획자가 편집하는 공간"이 아니게 된다.
그 시점부터 git의 원래 용도대로 쓸 수 있다.

| 현재 (SVN 방식) | 개선 후 (git 방식) |
|----------------|------------------|
| release 브랜치 = 데이터 편집 공간 | release 브랜치 = CI가 pb를 빌드하는 대상 (편집 없음) |
| 기획자가 브랜치 직접 전환하며 작업 | 기획자는 main에서만 작업 |
| 특정 버전 변경 이력 = 브랜치별 git log 각각 확인 | 변경 이력 = main의 git log 하나로 전체 추적 |
| "이 수치 누가 바꿨어?" = 추적 불가 | `git blame cookies.csv` 한 줄로 라인 단위 추적 |
| 기능 단위 작업 분리 불가 | feature 브랜치 → main PR 워크플로우 가능 |
| 동시 작업 = 구두 협의 필수 ("나 cookies.xlsx 작업해") | PR 단위로 충돌 시점 명확히 인지 가능 |

### 왜 지금 당장은 git답게 쓸 수 없는가

xlsx/pb가 바이너리인 한 git의 핵심 기능이 작동하지 않는다.

```
git diff cookies.xlsx  →  textconv로 텍스트 표시는 가능 (이미 설정됨)
                           그러나 모든 버전 행이 섞여 나옴, 버전별 필터링 불가
git blame cookies.xlsx →  커밋 단위 추적만 가능, 행 단위 추적 불가
git merge              →  xlmerge 있지만 불완전 (수식 깨짐)
```

**해결 순서:**
1. `datasheet --build` 옵션으로 단일 브랜치 전환 (본 문서 목표)
2. xlsx → CSV 중간 레이어 도입 (텍스트 변환 후 git에 저장)
3. CSV가 git에 저장되면 `git diff`, `git blame`이 행 단위로 작동

> Step 2~3은 중장기 과제다. datasheet가 CSV 입력을 지원해야 하며,
> 클라이언트/서버의 pb 로드 방식도 변경이 수반된다.
> 단, Step 1만 완성해도 기획자의 일상 작업 부담은 구조적으로 해소된다.

### CSV 도입 — git답게 쓰기의 핵심 전략

> **기획자 일상 업무와 직접 관련 없는 내용이다.**
> Phase 1~3 완성 이후, git을 제대로 활용하는 단계에서의 이야기다.
> 이 섹션의 내용은 추후 "git답게 쓰기" 별도 문서에서 상세히 다룬다.

#### CSV 있을 때 / 없을 때 비교

| 항목 | CSV 없는 현재 (xlsx 직접) | CSV 도입 후 |
|------|--------------------------|------------|
| `git diff` | textconv로 변경 내역 확인 가능 | CSV 텍스트 라인 단위 diff (textconv 불필요) |
| `git blame` | 커밋 단위 추적만 가능 | **행 단위 추적** — "이 수치 누가 언제 바꿨는지" |
| `git merge` | xlmerge (불완전, 수식 깨짐) | **완전한 텍스트 merge** — 라인 단위 자동 병합 |
| GitHub PR 리뷰 | diff 표시 안 됨 | 변경된 행을 PR 화면에서 바로 확인 |
| 개발 공수 | 낮음 (현재 상태) | 높음 (변환기, 정렬 로직, datasheet CSV 입력 지원) |
| 저장소 파일 수 | xlsx 129개 | xlsx 129개 + CSV ~261개 (시트당 1개 추가) |

#### 왜 지금은 불필요한가

Phase 1~3의 목표는 **"기획자가 main 브랜치 하나에서만 작업한다"** 는 것이다.
이 목표는 CSV 없이도 달성된다.

- `$filter` 컬럼 + `datasheet --build` 옵션 = 단일 브랜치 전환
- 기획자의 일상 작업(엑셀 수정 → 커밋)에는 CSV가 전혀 개입하지 않는다

CSV는 **"기획자가 어떻게 작업하느냐"가 아니라, "git이 변경 이력을 어떻게 추적하느냐"** 의 문제다.

#### CSV가 필요해지는 시점

```
단일 브랜치 운영이 안정화된 후:
  "이 수치 언제 바뀌었지?" → git blame으로 답이 나와야 한다
  "지난 달 패치에서 뭐가 바뀌었지?" → git diff로 한눈에 보여야 한다
  "두 명이 동시에 xlsx 수정했는데" → 자동 merge가 가능해야 한다
```

이 수요가 팀에서 실감되는 시점이 CSV 도입을 본격 논의할 때다.

> **결론: CSV 도입은 필요하지만 지금 당장은 아니다.**
> Phase 3(단일 브랜치 완전 전환) 안정 이후, git을 제대로 쓰는 단계에서 추진한다.
