# 파이프라인 방식 비교

## 배경: 근본 문제

**main 브랜치 = 언제든 배포 가능한 안정 상태** 가 git의 기본 원칙.
현재는 검증 게이트 없이 데이터가 바로 push되고 있다.

```
현재: xlsx 수정 → pb 생성 → 검증 없이 push → CI 오류 또는 dev에서 발견
이상: xlsx 수정 → 로컬 검증 → 통과 후에만 push → main 항상 안정
```

**결과:**
- CI 오류 누적 (실측 ~820건)
- dev 서버가 사실상 QA 환경으로 전락
- 잘못된 데이터가 전체 환경으로 전파 가능
- 슬랙 알람(#cos_noti_data_error)으로 팀 전체에 오류 공유 → 수정될 때까지 push 불가

**pre-push hook으로 해결 (Step 1):**
```
현재: push → CI 실패 → 슬랙 알람 → 팀 전체 차단

개선: push 시도 → datasheet.exe -strict → 로컬에서 즉시 차단
```
CI와 동일한 에러 메시지(FK 오류, 스키마 오류 등)를 로컬에서 먼저 출력.
슬랙 알람 오기 전에 본인이 먼저 인지하고 수정 가능.

## 검증 레이어

| 검증 레이어 | 도구 | 현재 상태 |
|------------|------|---------|
| 1차: 스키마/타입 오류 | datasheet.exe | ✅ 로컬 가능 |
| 2차: pb-서버 호환성, 데이터 로드 | 로컬 서버 | ❌ 팀에서 미사용 |
| 3차: 게임 논리/인터랙션 | dev 서버 | 현재 유일한 검증 환경 |

> **2차 검증 부재의 문제**: 스키마는 맞아도 서버 기동 시 문제 발생 가능.
> 로컬 서버 = push 전 마지막 게이트. 단, 아이템 구매 등 인터랙션은 다중 서비스 의존 → dev 서버에서만 가능.
> 로컬 서버 기획자 접근성 개선은 **별도 개발팀 협의 과제 (Step 2.5)**.

## 3가지 방식 정의

| | 방식 1 | 방식 2 | 방식 3 |
|--|--------|--------|--------|
| **흐름** | xlsx → 로컬 pb → push xlsx → CI pb | xlsx → CSV → push 둘다 → CI pb | xlsx → push xlsx → CI pb |
| **로컬 빌드** | datasheet.exe (pb 생성) | datasheet.exe (CSV 생성) | 없음 |
| **git push 대상** | xlsx만 | xlsx + CSV | xlsx만 |
| **pb 생성 주체** | CI | CI (CSV → pb 변환) | CI |

> **방식 2에서 로컬 pb의 역할:**
> - 로컬 테스트용으로 기획자가 직접 생성 (datasheet.exe로 CSV → pb 변환)
> - Unity/서버가 로컬에서 pb를 읽어 테스트 → 코드 변경 불필요
> - **git에는 절대 push 금지** → `.gitignore`에 pb 추가로 구조적으로 차단
> - git의 소스 오브 트루스는 CSV, pb는 빌드 결과물

## 항목별 비교

### 1차 기술 검증 vs 2차 논리 검증

| 방식 | 1차: 스키마/타입 오류 | 2차: pb-서버 호환성 | 3차: 게임 논리 |
|------|-------------------|------------------|-------------|
| 방식 1 | ✅ 로컬 pb 생성 시 즉시 감지 | 🔶 로컬 서버 활용 시 가능 | 🔶 dev 서버 필요 |
| 방식 2 | ✅ 동일 | 🔶 동일 | 🔶 dev 서버 필요 |
| 방식 3 | 🔶 pre-push hook (schema only) | ❌ CI 실패 후에야 감지 | ❌ CI 실패 후에야 감지 |

> 방식 1도 "dev에서 논리 검증" 문제는 그대로. 방식 1의 실질 장점은 **1차 오류 조기 감지 + 오류 전파 범위 감소**.

### 오류 전파 범위 (핵심)

**현재 수동 전파 구조의 숨겨진 장점:**
- main → (확인) → h2 → (확인) → h3 순서로 전파 → 자연스러운 스테이징 버퍼
- main에 잘못된 데이터 push 후 발견 → h2/h3 전파 전에 차단 가능

| 방식 | 오류 전파 범위 | 설명 |
|------|-------------|------|
| 방식 1 | 🟢 낮음 | 로컬 1차 검증 통과 후 push. CI 실패해도 이미 로컬 검증 완료 |
| 방식 2 | 🟢 낮음 | 방식 1과 동일 |
| 방식 3 | 🔴 **치명적** | push 즉시 CI가 h2/h3/vng 전체 동시 배포 → 오류 시 모든 환경 동시 오염. CI 실패 동안 전체 작업자 push 불가 |

### git 운영

| 방식 | diff | merge | cherry-pick |
|------|------|-------|------------|
| 방식 1 | 🔶 Fork textconv로 가능 | ❌ 바이너리 한계 | ❌ |
| 방식 2 | ✅ CSV 네이티브 | ✅ CSV 네이티브 | ✅ CSV 네이티브 |
| 방식 3 | 방식 1과 동일 | ❌ | ❌ |

### 개발 비용 / 기획자 변화 / 저장소

| | 방식 1 | 방식 2 | 방식 3 |
|--|--------|--------|--------|
| **개발 비용** | 🔶 중간 — CI 파이프라인 변경 | 🔴 높음 — CSV 출력 + CI CSV→pb 변환 추가 | 🟢 낮음 — pre-push hook만 |
| **기획자 실무 변화** | 작음 — datasheet.exe 유지, 커밋 대상만 변경 | 중간 — CSV 추가 관리 | 매우 작음 — 바로 push |
| **저장소 크기** | 감소 — pb 제거 | 현상 유지 이상 — xlsx + CSV | 감소 — pb 제거 |
| **동기화 리스크** | 없음 | 있음 — xlsx:CSV 불일치 가능 | 없음 |

## 권고안

### 방식 3 제외
- 로컬 검증 없음 + 전체 환경 동시 오염 → **현재 수동 전파보다 리스크가 큼**
- pre-push hook(schema only)으로는 논리/호환성 오류 사전 차단 불가
- 한 번 오류 시 팀 전체 작업 중단 위험

### 방식 2 — Step 3 목표로 재정의

**핵심 인사이트:** CSV가 git의 소스 오브 트루스가 되면 바이너리 한계를 우회하면서 git 브랜치 전략을 완전히 활용 가능.

```
xlsx → CSV (git 추적) → CI가 CSV → pb 변환 → 서버 배포
                  ↓
         로컬: CSV → pb (테스트용, .gitignore로 push 차단)
                  ↓
         Unity/서버가 로컬 pb로 테스트 (클라이언트 코드 변경 불필요)
```

| git 기능 | 방식 1 (xlsx) | 방식 2 (CSV) |
|---------|-------------|-------------|
| diff | 🔶 textconv | ✅ 네이티브 |
| merge | ❌ | ✅ 서로 다른 행 수정 시 자동 |
| cherry-pick | ❌ | ✅ |
| blame | ❌ | ✅ |
| PR 리뷰 | ❌ 수치 확인 불가 | ✅ 변경 수치 텍스트로 검토 |

**보류 → Step 3 채택 조건:**
- datasheet.exe CSV 출력 지원 (개발팀 작업)
- CI CSV → pb 변환 파이프라인 구축
- pb `.gitignore` 추가 (로컬 pb push 구조적 차단)
- xlsx:CSV 동기화 강제화 (hook 또는 툴)

### 방식 1 채택 (Step 2 목표)

**흐름:**
```
기획자: xlsx 수정 → datasheet.exe → (로컬 서버 확인) → xlsx만 push
CI:     push 감지 → datasheet --build <VERSION> → 버전별 pb 생성/배포
```

**효과:**
- 1차 기술 오류 로컬에서 조기 차단
- pb git에서 제거 → 저장소 크기 감소
- 수동 브랜치 전파 제거
- 오류 전파 범위 감소

## 로드맵과의 연결

| 단계 | 목표 | 검증 게이트 |
|------|------|-----------|
| Step 1 | 커밋 컨벤션 + pre-push hook | 1차 (schema) |
| Step 2 | 방식 1 채택 + $filter + CI pb 자동화 | 1차 (datasheet.exe 로컬) |
| Step 2.5 | 로컬 서버 기획자 접근성 개선 (개발팀 협의) | 1차 + 2차 (로컬 서버) |
| Step 3 | 방식 2 채택 — CSV 도입 + feature branch + PR | 1차 + 2차 + diff/merge/PR 리뷰 |

> **Step 3가 진짜 "git답게 쓰기"인 이유:**
> CSV가 소스 오브 트루스가 되면 xlsx 바이너리 한계를 우회하여 diff/merge/cherry-pick/PR 리뷰 모두 가능.
> pb는 로컬 테스트용으로만 생성, `.gitignore`로 push 차단, CI가 CSV 기반으로 자동 생성.
