# 🚀 기획 데이터 버전 관리 및 배포 파이프라인 (Git & CI/CD)

본 문서는 기획 데이터(Excel)의 로컬 작업부터 Git 버전 관리, 그리고 Jenkins를 통한 다중 버전(h1, h2, h3) 빌드 및 배포까지의 전체 워크플로우를 정의합니다.

---

## 1. 파이프라인 핵심 원칙
* **단일 브랜치 (Single Branch):** 기획 데이터는 오직 `main` 브랜치에서만 관리 및 병합(Merge)됩니다.
* **텍스트 추적 (Text-based Tracking):** Git은 바이너리 파일(`.pb`, `.xlsx`)을 추적하지 않으며, 오직 `.csv` 파일만 추적하여 충돌(Conflict) 발생 시 라인 단위로 해결합니다.
* **단일 진실 공급원 (Single Source of Truth):** Unity 클라이언트는 오직 바이너리(`.pb`) 포맷만 읽습니다. CSV 파싱은 파이프라인(Python 스크립트)에서 전담합니다.

---

## 2. 데이터 구조 및 버전 통제 (Override 기법)
데이터의 버전별 분기는 브랜치가 아닌, 데이터 내부의 **구간(Range) 명시 방식**을 통해 통제합니다. 

### 2.1. 필수 컬럼 정의
모든 기획 데이터 시트에는 다음 두 개의 컬럼이 반드시 포함되어야 합니다.
* `Min_Version`: 데이터가 적용되기 시작하는 최소 빌드 버전
* `Max_Version`: 데이터 적용이 끝나는 최대 빌드 버전 (영구 적용 시 `999` 등 최대치 사용)

### 2.2. 데이터 작성 예시 (`Character_Stats.xlsx`)
| ID | Name | Attack | Min_Version | Max_Version | 비고 (기획자 참고용) |
| :--- | :--- | :--- | :--- | :--- | :--- |
| 1001 | 슬라임 | 10 | 1 | 999 | 기본 몬스터 (전체 버전) |
| 1002 | 고블린 | 20 | 2 | 999 | h2(Target=2) 추가 몬스터 |
| **2001** | **보스** | **100** | **3** | **3** | **h3(Target=3) 이벤트 전용 보스** |

---

## 3. 기획자 로컬 작업 워크플로우

기획자는 Git 브랜치 이동 없이 `main` 환경에서 엑셀 툴링을 통해 즉각적인 로컬 테스트를 수행합니다.

### 3.1. 엑셀 원클릭 빌드 (VBA 매크로)
엑셀 내부에 `[CSV 저장 및 로컬 PB 빌드]` 버튼을 배치하여 다음 작업을 백그라운드에서 자동화합니다.
1. 현재 시트 데이터를 Git 추적 대상 폴더(`/Tracked_CSV`)에 `.csv`로 저장
2. 파이썬 빌드 스크립트 호출: `python csv_to_pb.py --target={로컬테스트버전}`
3. 생성된 `.pb` 파일을 Unity 에셋 경로(`Assets/Resources/Data` 등)로 덮어쓰기

### 3.2. Unity 클라이언트 테스트
1. 엑셀에서 버튼 클릭 시 즉시 Unity로 데이터가 밀어넣어집니다.
2. Unity 에디터를 클릭하여 Asset Refresh를 유도한 후 플레이 버튼을 눌러 변경된 스펙을 확인합니다.

---

## 4. Git 형상 관리 (Fork)

로컬 테스트가 끝난 데이터는 Git GUI 툴(Fork)을 통해 서버로 전송됩니다.

1. **Pull:** `main` 브랜치의 최신 변경 사항을 받아옵니다.
2. **Diff 확인:** `Tracked_CSV` 폴더 내 변경된 `.csv` 파일의 내역을 라인 단위로 확인합니다.
3. **Commit & Push:** 바이너리(`.pb`, `.xlsx`)가 포함되지 않았는지 확인 후, `.csv`만 `main` 브랜치에 푸시합니다.
 *(※ 다른 작업자와 동일한 라인을 수정하여 Conflict가 발생한 경우, Fork 환경에서 텍스트 Merge를 통해 즉시 해결합니다.)*

---

## 5. Jenkins CI/CD 배포 파이프라인

`main` 브랜치에 새로운 `.csv`가 푸시되면 Jenkins가 이를 감지하여 모든 라이브 및 개발 버전의 데이터를 생성하고 배포합니다.

### 5.1. 다중 버전 동시 빌드 (Regenerate All)
Jenkins는 내부 빌드 스크립트를 통해 현재 운영/개발 중인 모든 타겟 버전에 대해 PB 파일을 굽습니다. `pandas` 데이터프레임을 활용하여 `Min_Version <= Target <= Max_Version` 조건에 맞는 행만 추출하여 직렬화합니다.

* `python csv_to_pb.py --target=1` ➔ `h1_char.pb` 생성 (라이브용)
* `python csv_to_pb.py --target=2` ➔ `h2_char.pb` 생성 (다음 패치용)
* `python csv_to_pb.py --target=3` ➔ `h3_char.pb` 생성 (다다음 패치용)

### 5.2. 해시 비교 및 CDN 배포
생성된 `.pb` 파일들의 무의미한 덮어쓰기를 방지하기 위해 배포 스크립트가 개입합니다.
1. 새로 구워진 `.pb` 파일의 해시값(MD5 등)을 추출합니다.
2. 기존 CDN(또는 패치 서버)에 올라가 있는 버전의 해시값과 비교합니다.
3. **해시값이 변경된 파일만** 각 버전별 전용 경로(`/CDN/Live_h1/`, `/CDN/Dev_h2/` 등)로 업로드합니다.
