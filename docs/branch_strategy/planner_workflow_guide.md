# 기획자 실무 가이드
## cos-data git 작업 방식 — Step별 상세 안내

> 기준일: 2026-04-07
> 대상: cos-data 기획팀 전원

---

## Step 1 — 지금 당장 적용 (개발팀 도움 불필요)

### 1-1. 커밋 메시지 작성법

**형식:**
```
[버전태그] 작업 내용 [Wrike:번호]
```

**버전 태그 기준:**

| 태그 | 언제 쓰는가 |
|------|------------|
| `[H3]` | H3 버전 전용 작업 |
| `[H2]` | H2 라이브 핫픽스 |
| `[VNG]` | VNG 전용 작업 |
| `[ALL]` | 모든 버전에 공통 적용 |
| `[HF]` | 긴급 핫픽스 (버전 무관) |
| `[Test]` | 임시 테스트 데이터 (나중에 원복 예정) |

**예시:**

```
✅ [H3] 까망베르 공격력 밸런스 최종 [Wrike:4421234]
✅ [ALL] 코인사용량 퀘스트 반영 [Wrike:4428567]
✅ [HF] 광장 쿨타임 버그 수정 [Wrike:4430001]
✅ [Test] 까망베르 공격력 90% 감소 테스트

❌ [추태영] 까망베르 밸런스 4차
❌ 밸런스 수정
❌ [H3][추태영] 작업내용
```

**Wrike 번호 찾는 법:**
- Wrike 태스크 URL: `app.wrike.com/open.htm?id=XXXXXXXX` → 숫자 부분
- 또는 태스크 상세 페이지 우측 상단 ID

### 1-2. pre-push hook 설치 (최초 1회)

push 전에 자동으로 데이터 유효성 검사를 실행합니다.

**Git Bash 또는 Fork 터미널에서 실행 (cos-data 루트):**
```bash
cp .git/hooks/pre-push.sample .git/hooks/pre-push
chmod +x .git/hooks/pre-push
```

`.git/hooks/pre-push` 파일 내용:
```bash
#!/bin/sh
./datasheet.exe -strict
if [ $? -ne 0 ]; then
  echo "데이터 검증 실패. push가 차단됩니다."
  exit 1
fi
```

> Git for Windows(Git Bash) 환경에서 동작합니다. Fork, SourceTree 등 GUI 툴도 내부적으로 Git Bash를 사용하므로 정상 작동합니다.

**설치 확인:** push 시도 시 `datasheet.exe -strict` 실행 메시지가 보이면 정상.

### 1-3. [Test] 태그 원칙

- 테스트 커밋을 push했다면 → 원복 커밋도 반드시 `[Test]` 태그 포함
- 가능하면 push 전 로컬에서 원복 후 정리
- 테스트 완료 후 `git log --grep="\[Test\]"`로 테스트 커밋 확인 가능

## Step 2 — 개발팀 협의 후 적용

> 이 단계는 `datasheet --build` 옵션 구현 완료 후 전환합니다.
> 전환 공지가 있기 전까지는 Step 1 방식 유지.

### 2-1. 기본 원칙

- **브랜치는 main 하나만** 사용 (release_* 브랜치 종료)
- **pb 파일은 커밋하지 않음** (CI가 자동 생성)
- **xlsx만 커밋**

### 2-2. 일상 작업 흐름

```
1. main 브랜치 pull (최신 상태 동기화)
2. Excel 파일 수정
3. $filter 컬럼 확인 (어느 버전에 반영될 것인지)
4. Fork에서 diff 확인 (xlsx 변경 내역만) — pb 파일이 목록에 없으면 정상
5. 커밋 메시지 작성: [H3] 내용 [Wrike:번호]
6. push → CI가 자동으로 버전별 pb 생성 및 배포 (GitHub Actions에서 진행 상황 확인 가능, 약 3~5분 소요)
7. CI 완료 후 해당 버전 dev 서버 접속 → 확인
```

### 2-3. $filter 컬럼 작성 기준

| 상황 | 사용할 $filter |
|------|---------------|
| 모든 버전에 적용 | `$$LAUNCH_0` |
| H3 이상 전체 적용 | `$$HELSINKI_3` |
| IST1 이상 전체 적용 | `$$ISTANBUL_1` |
| VNG 전용 | `$$VNG_ONLY` |
| H3에만 적용, IST1 제외 | `$$HELSINKI_3_oneTarget` *(개발팀 협의 완료 전까지 사용 보류)* |

**예시 — 까망베르 HP 버전별 조정:**

| id | hp | $filter | 설명 |
|----|-----|---------|------|
| 1000 | 100 | `$$LAUNCH_0` | 베이스 (H2 이하) |
| 1000 | 120 | `$$HELSINKI_3` | H3 이상 HP=120 |
| 1000 | 150 | `$$ISTANBUL_1` | IST1 이상 HP=150 |

### 2-4. 버전별 dev 서버 확인

| 수정 대상 | 확인할 서버 |
|----------|------------|
| H3 관련 수정 | H3 dev 서버 |
| VNG 관련 수정 | VNG dev 서버 |
| 전 버전 공통 | H3 dev 서버 (가장 최신) |

## Step 3 — git을 git답게 (추후 안내 예정)

이 단계는 Step 2 안정화 후 팀 전체 논의를 거쳐 전환합니다.

**변화 예고:**
- 공유 브랜치 직접 push → **개인 feature branch 생성 후 PR**
- 테스트 데이터 실험 → **개인 브랜치에서 격리 후 확인, PR로 merge**
- 커밋 히스토리로 "왜 바뀌었는지" 추적 가능

## 자주 묻는 질문

**Q. Wrike 번호를 꼭 넣어야 하나요?**
A. Step 1에서는 권장 사항입니다. Step 2부터는 필수로 운영할 예정입니다.
번호가 없으면 나중에 "왜 이 수치가 바뀌었는지" 추적이 어렵습니다.

**Q. [ALL] 태그는 언제 쓰나요?**
A. 모든 버전(H2, H3, VNG 등)에 동시에 반영되어야 하는 수정일 때 씁니다.
확실하지 않으면 [H3]처럼 특정 버전 태그를 쓰는 게 안전합니다.

**Q. Step 2 전환 전까지 $filter 컬럼을 미리 써도 되나요?**
A. 네, xlsx에 $filter 컬럼을 미리 작성해 두어도 무방합니다.
현재 datasheet는 $filter 컬럼을 인식하지 않으므로 게임에 영향 없습니다.

**Q. pb 파일을 실수로 커밋했어요.**
A. Step 1 시점에는 pb 커밋이 정상입니다. Step 2 전환 후 pb 커밋이 발생하면
팀 공지 채널에 알려주세요.
