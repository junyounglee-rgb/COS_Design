## Slide 1 — 기(起): 지금 우리의 상황

### 현재 구조의 문제

- git을 SVN처럼 쓰면서 기획 데이터를 **여러 브랜치로 관리**하고 있다
- 같은 수정을 브랜치 수만큼 반복 → **휴먼 에러가 나기 좋은 구조**
- 라이브 데이터 변경이 빈번한데, 지금 구조로는 **기민하게 대처하기 어렵다**
- 커밋 메시지 규칙이 없어 **"왜 바꿨는지" 히스토리 추적이 어렵다. 이번에 H2에서 비슷한 이슈가 생겼음**

### 실측 수치 (4개월)

| 지표 | 수치 |
|------|------|
| CI 오류 | ~820건 (일 평균 6.7건) |
| 평균 수정 소요 | 37분 (최대 127분) |
| PR 사용률 | 1.5% — SVN식 운영에서는 PR을 쓸 상황 자체가 없음 |
| `.git` 크기 | 4.6GB — pb 바이너리 누적 |

## Slide 2 — 승(承): 왜 SVN처럼 쓸 수밖에 없었나

### 입력도 바이너리, 출력도 바이너리

| 파일 | 역할 | 형식 |
|------|------|------|
| `.xlsx` | 기획자가 편집하는 데이터 원본 | **바이너리** |
| `.pb` | 서버/클라이언트가 읽는 빌드 결과물 | **바이너리** |

### 바이너리가 강제하는 구조

```
xlsx, pb = 바이너리
    │
    ├─► merge/cherry-pick 불가
    │       → 버전마다 파일을 통째로 분리해야 함
    │       → 브랜치 = 파일 복사본 (SVN의 release branch)
    │
    └─► xlsx와 pb가 항상 쌍으로 동기화되어야 게임 작동
            → pb 빠뜨리면 CI 차단 → 팀 전체 push 불가
```

> 지금의 SVN식 운영은 팀의 선택이 아니라,
> **바이너리 파일 구조가 강제한 필연적 결과**

## Slide 3 — 승(承): 지금 구조의 실체

### 같은 수정을 브랜치 수만큼 반복

```
기획자 A — cookies.xlsx 수정 (release_helsinki3)
기획자 A — cookies.xlsx 수정 (release_helsinki2)   ← 동일 작업 반복
기획자 A — cookies.xlsx 수정 (release_helsinki_vng) ← 또 반복
기획자 A — cookies.xlsx 수정 (main)                 ← 또 반복
```

브랜치가 늘수록 반복 횟수도 늘어난다.
해외 퍼블리셔가 추가되면 5개, 6개가 될 수 있다.

### 오류 있는 push → 오류가 전체에 전파

```
기획자 A — 검증 없이 push
    ↓
CI 차단 (xlsx:pb not synced / 스키마 오류 등)
    ↓
기획자 B, C, D ... 도 push 불가
    ↓
A가 수정할 때까지 전체 대기
```

**근본 문제:** main 브랜치에 검증되지 않은 데이터가 올라간다.
검증 게이트가 없으니 dev 서버가 사실상 QA 환경으로 전락한 상태.

## Slide 4 — 전(轉): 해결 방향

### xlsx — $filter 컬럼으로 main 단일 브랜치 관리

브랜치로 버전을 나누는 대신, **파일 안에서 버전을 구분**한다.

| id | hp | $filter | 결과 |
|----|-----|---------|------|
| 1000 | 100 | `$$LAUNCH_0` | 모든 버전 HP=100 |
| 1000 | 120 | `$$HELSINKI_3` | H3 이상 HP=120 |
| 1000 | 150 | `$$ISTANBUL_1` | IST1 이상 HP=150 |

> `$filter` 설계는 `keywords.xlsx`에 이미 구현되어 있다.

### pb — git에서 꺼낸다, CI가 자동 생성

| 항목 | 현재 | 목표 |
|------|------|------|
| pb 생성 | 기획자 로컬 수동 실행 | **CI가 자동 생성** |
| pb 커밋 | 기획자가 직접 포함 | **git 추적 대상에서 제거** |
| 버전별 빌드 | 브랜치로 분리 | `datasheet --build HELSINKI_3` 등 |

### 파이프라인 방식 비교 (핵심)

| 방식 | 1차: FK/스키마 오류 | 2차: 서버 로드 오류 | 오류 전파 범위 | 개발 비용 | 권고 |
|------|------------------|------------------|-------------|---------|------|
| 방식 1: xlsx→로컬pb→push xlsx→CI pb | ✅ 로컬 (`-strict`) | 🔶 로컬 서버 있어야 가능 | 🟢 낮음 | 🔶 중간 | **채택** |
| 방식 2: xlsx→CSV→push 둘다→CI pb | ✅ 로컬 (`-strict`) | 🔶 로컬 서버 있어야 가능 | 🟢 낮음 | 🔴 높음 | Step 3 목표 |
| 방식 3: xlsx→push→CI pb | 🔶 pre-push hook만 | ❌ CI 실패 후에야 감지 | 🔴 **전체 동시 오염** | 🟢 낮음 | **제외** |

> **방식 1 채택 시 검증 흐름:** 로컬 pb 생성 단계에서 `-strict` 옵션으로 FK/스키마 오류를 차단한다.
> push hook에서 datasheet를 재실행하지 않아도 되는 이유: 팀 규칙상 push 전 pb 생성이 보장되므로 이미 검증 완료 상태다.
> **로컬 서버가 없으면:** 1차(FK/스키마)는 막을 수 있지만, pb가 서버에서 정상 로드되는지는 dev 서버 의존 끊을 수 없음.
> **로컬 서버가 있어야** push 전 완전한 검증 가능 → **Step 2.5(로컬 서버 접근성 개선)가 핵심 과제**.
> 방식 3 제외 이유: push 즉시 h2/h3/vng 전체 동시 배포 → 오류 시 팀 전체 작업 중단.

## Slide 5 — 전(轉): 지금 당장 시작할 수 있는 것 — Step 1

> 개발팀 도움 없이, 툴 변경 없이 오늘부터 적용 가능

### git 로그 규칙 정하기

```
현재:  [이준영] 까망베르 밸런스 4차
개선:  [H3] 까망베르 밸런스 최종 [Wrike:4421xxx]
```

| 태그 | 의미 |
|------|------|
| `[H3]` `[H2]` `[VNG]` | 버전 전용 |
| `[ALL]` | 전 버전 공통 |
| `[H2-HF]` | h2버전의 핫픽스 |
| `[Test]` | 임시 테스트 데이터 (원복 예정) |

### pre-push 검증 hook

**hook이란?** Fork에서 push 버튼을 누르는 순간 자동으로 실행되는 검사 스크립트.
통과해야만 push가 진행되고, 실패하면 서버에 올라가기 전에 로컬에서 차단된다.

```
현재: push → CI 실패 → 슬랙 알람 → 팀 전체 차단    ← 이미 늦음

개선: push 시도
        → 커밋 메시지 태그 형식 검사 ([H3], [ALL], [Test] 등 누락 시 차단)
        → 변경된 xlsx 파일 목록 출력 (변경 범위 인지)
        → push 진행
```

**커밋 메시지 태그 형식 (hook 검사 대상):**
```
통과:  [H3] 까망베르 밸런스 최종 [Wrike:4421xxx]
통과:  [ALL] 공통 데이터 수정
통과:  [Test] 임시 테스트 데이터
차단:  [이름] 까망베르 밸런스 4차   ← 버전 태그 없음
```

### 레거시 브랜치 정리

방치된 브랜치 7개 (release-essen, release-fargo, release-geneva 등) → 태그 후 삭제

## Slide 6 — 전(轉): 실무는 어떻게 바뀌는가(단기 목표 Step 2)

### Step 1 — 지금 당장(ALL)

| 항목 | 현재 | 변화 |
|------|------|------|
| 커밋 메시지 | `[이름] 작업내용` | `[H3] 내용 [Wrike:번호]` |
| push 검증 | CI 실패 후 알람 | **pre-push hook 로컬 자동 차단** |
| 브랜치 | release_* (변화 없음) | — |
| pb 커밋 | xlsx + pb 함께 (변화 없음) | — |

### Step 2 — 1차 개선

| 항목 | 현재 | 변화 |
|------|------|------|
| 브랜치 | release_helsinki3 / 2 / vng | **main 하나만** |
| 커밋 대상 | xlsx + pb | **xlsx만** |
| 버전 지정 | 브랜치로 분리 | **$filter 컬럼으로** |
| pb 생성 | 로컬 datasheet.exe 수동 | **CI 자동 생성** |

> 같은 수정을 N번 반복하는 구조가 사라진다.

### Step 2.5 — 로컬 검증 강화

| 항목 | 현재 | 변화 |
|------|------|------|
| 로컬 서버 | 미사용 | **기획용 로컬 서버 구축 (개발팀 협의)** |
| 검증 범위 | 1차 스키마만 | **2차 pb 로드 + 기본 게임 데이터 동작** |
| 실행 방법 | — | **Unity DEV 메뉴 → local 클릭 (서버 자동 기동)** |

**범위:**
- 필요한 것: pb 로드 확인, 기본 게임 데이터 동작 확인
- 개발이 어려운 것: 인게임내 서버 통신(아이템 구매, 결제 등 외부 서비스 연동)

### Step 3 — git을 git답게 (CSV 도입, 프로젝트 적용은 그대로 .pb)

| 항목 | 현재 | 변화 |
|------|------|------|
| git 소스 | xlsx (바이너리) | **CSV (텍스트)** |
| 로컬 테스트 | pb 생성 후 Unity/서버 | **동일 — CSV → 로컬 pb → 테스트** |
| 로컬 pb push | 실수로 가능 | **`.gitignore`로 구조적 차단** |
| diff/merge | ❌ 바이너리 한계 | **✅ CSV 네이티브** |
| PR 리뷰 | 수치 확인 불가 | **변경 수치를 텍스트로 검토 가능** |
| 작업 방식 | 공유 브랜치 직접 push | **feature branch → PR → merge** |

> xlsx, pb merge는 여전히 안 되지만, CSV가 소스 오브 트루스가 되면
> git 브랜치 전략을 완전히 활용할 수 있다.

## Slide 7 — 결(結): 논의 필요 사항

### 팀 내 룰 결정 (Step 1 시작 전)

| 항목 | 내용 |
|------|------|
| 버전 태그 형식 통일 | `[H3]` / `[Helsinki3]` / `[h3]` 중 하나로 |
| `[Test]` 태그 운영 | 테스트 커밋 원복 방식 및 범위 |
| pre-push hook 배포 | 팀원 전체 설치 방법 및 시점 |

### $$TEST 필터 운영 룰 (Step 2 시점)

main 단일 브랜치에서 테스트 데이터를 안전하게 격리하는 방법.

```
keywords.xlsx 추가:
TEST = 9999999998  ← DEV(9999999999)보다 낮게 설정
```

| $filter 값 | 포함 환경 |
|-----------|---------|
| `$$LAUNCH_0` ~ `$$HELSINKI_3` | h2 / h3 / vng / main 전체 |
| `$$TEST` | **main dev 환경만** (h2/h3/vng 절대 미포함) |

**운영 방식:**
- 테스트 데이터 행에 `$$TEST` 필터 적용
- 커밋 메시지에 `[Test]` 태그 병행 사용
- 테스트 완료 후 해당 행 삭제로 정리 끝

**장점:**
- 별도 브랜치/서버 불필요
- 테스트 데이터가 실수로 h2/h3/vng에 배포될 위험 없음 (CI가 자동 차단)
- main dev에서 확인 후 `$$HELSINKI_3` 등으로 필터 변경하면 정식 반영

### 개발팀 협의 필요 (Step 2 착수 전)

| 항목 | 내용 |
|------|------|
| `datasheet --build` 구현 | 버전별 pb 자동 생성 옵션 |
| CI 파이프라인 변경 | xlsx push → CI 버전별 pb 자동 생성/배포 |
| pb `.gitignore` 처리 | pb를 git 추적 대상에서 제외 |
| 로컬 서버 기획자 접근성 개선 | push 전 2차 검증 게이트 확보 — 아래 옵션 비교 참고 (Step 2.5) |
| 저장소 히스토리 정리 | 기존 pb 커밋 처리 방법 |

### Step 2.5 — 로컬 검증 환경 옵션 비교

push 전 2차 검증(pb가 서버에서 정상 로드되는가)을 위한 두 가지 방법.

| | 테스트 브랜치 + CI 배포 | 로컬 서버 |
|--|----------------------|---------|
| 방법 | cos-data에 test 브랜치 → CI가 test 서버에 배포 | 기획자 PC에서 직접 서버 실행 |
| 인프라 | 테스트 서버 환경 추가 필요 | 없음 |
| 개발팀 작업 | CI 파이프라인 수정 | 로컬 실행 환경 정비 |
| 피드백 속도 | CI 대기 시간 발생 | 즉시 확인 |
| 동시 사용 | ❌ 여러 명이 같은 test 브랜치 공유 → 충돌 | ✅ 각자 로컬에서 독립 실행 |

**테스트 브랜치가 장기적으로 좋지 않은 이유:**
- 지금의 h2/h3/vng 브랜치와 동일한 문제 반복 — 브랜치가 하나 더 늘어나는 것
- 여러 명이 동시에 테스트하면 서로의 데이터가 섞임
- CI 배포 기다리는 동안 병목 발생 → 결국 "dev 서버에서 확인"과 큰 차이 없음
- Step 2의 목표(브랜치 줄이기)와 방향이 반대

> **권고: 로컬 서버** — 브랜치 없이 각자 독립적으로 즉시 검증 가능.
> 범위: pb 로드 확인 + 기본 게임 데이터 동작. 구매/결제 등 외부 서비스 연동 불필요.

**로컬 서버 구현 방법 (이미 DEV 메뉴 인프라 존재):**

Unity DEV 메뉴에 이미 서버 선택 UI가 있음 → `local` 버튼 하나 추가하면 끝.

```
Unity DEV 메뉴 → local 버튼 클릭
    ↓
(자동) 로컬 서버 프로세스 실행
(자동) localhost 접속
    ↓
CMD 창에서 서버 로그 확인
```

**기획자 최종 워크플로우:**
```
1. xlsx 수정
2. datasheet.exe -strict → pb 생성
3. Unity DEV 메뉴 → local 클릭 (서버 자동 기동 + 접속)
4. CMD 창에서 에러 확인 / 게임에서 데이터 확인
5. 문제 없으면 push
```

**에셋 로딩:**
- 서버 환경 선택 시 해당 환경의 CDN 에셋을 로드하는 구조
- `local` 선택 시 → dev CDN 에셋 사용으로 설정하면 됨 (데이터 테스트 목적엔 충분)
- 에셋까지 로컬로 내려받는 건 불필요 (데이터와 에셋은 별개)
- ⚠️ 정확한 에셋 로딩 구조는 개발팀 확인 필요

**개발팀 요청 항목 (Step 2.5):**
- Unity DEV 메뉴에 `local` 버튼 추가 (클릭 시 서버 자동 기동 + localhost 접속)
- 로컬 서버 실행 시 dev CDN 에셋 사용하도록 설정
- 에러 로그 해석 가이드 (자주 나오는 에러 메시지 정리)

## Slide 8 — 결(結): 점진적으로 git을 git답게

### 로드맵

| Step | 목표 | 내용 | 담당 |
|------|------|------|------|
| **Step 1** | git 로그 규칙 정하기 | 커밋 컨벤션 + pre-push hook + 레거시 정리 | 기획팀 자체 |
| **Step 2** | main 브랜치 단일 작업 전환 | 방식 1 채택 + $filter + CI pb 자동화 | **개발팀 협의** |
| **Step 2.5** | 로컬 검증 강화 | 로컬 서버 기획자 접근성 개선 | **개발팀 협의** |
| **Step 3** | git을 git답게 쓰기 | CSV 도입 → 소스 오브 트루스 전환 + feature branch + PR | **중/장기적 목표** |

### Step 3 완료 시 변화

| 항목 | 현재 | Step 3 |
|------|------|--------|
| 활성 브랜치 수 | 4개 | **1개 (main)** |
| 같은 수정 반복 | N번 (브랜치 수만큼) | **1번** |
| git 소스 오브 트루스 | xlsx (바이너리) | **CSV (텍스트)** |
| pb 커밋 | 필수 (수동) | **없음 — 로컬은 테스트용만, CI가 자동 생성** |
| 로컬 pb push | 실수 가능 | **`.gitignore`로 구조적 차단** |
| diff / merge / cherry-pick | ❌ 바이너리 한계 | **✅ CSV 네이티브** |
| 로컬 검증 게이트 | 없음 | **로컬 1차 + 로컬 서버 2차** |
| CI 오류 전파 | 팀 전체 차단 | **로컬에서 사전 차단** |

### 핵심 요약

> 바이너리 구조 때문에 SVN처럼 쓸 수밖에 없었다.
> Step 2: xlsx는 $filter로, pb는 CI로 + 로컬서버 사용 — 브랜치 전파 문제 해결
> Step 3: CSV를 소스 오브 트루스로 — 진짜 git 브랜치 전략 활용 가능(훗날 과제)
> Step 1(지금) → Step 2(개발팀과 함께) → Step 3(git답게)
