# COS L10N 파이프라인 개선 제안

> 작성일: 2026-04-04
> 분석 대상: cos-data 번역 워크플로우 전반
> 목적: 수작업 중심 번역 파이프라인 → 자동화 파이프라인 전환

## 1. 현재 상황과 담당자 고충

### 담당자 현황

- **번역 파이프라인 전담: 1명 (PM)**
- 이 작업은 PM의 본래 업무(기획, 일정 관리, 커뮤니케이션)와 완전히 별개의 성격
- 별도 전문성 없이 수행 중인 **반복 수작업**이며, 게임 출시/업데이트 주기마다 반복

### 담당자가 매번 수행하는 작업 (1회 기준)

```
1. Fork 실행 → cos-data 최신화
2. Fork → Repository → "번역키 추출 (cos-common)" 버튼 클릭
3. 20초 대기 → exports/ 폴더에 타임스탬프 파일 생성 확인
4. 추출 파일을 7개 카테고리로 수동 분류
   (dialog / cookie / quests / ui / item / skill / etc)
5. lokalise.com 접속 → 프로젝트 선택
6. 카테고리별 파일을 각각 업로드 (설정값 매번 수동 입력)
   - Key → Key 매핑 설정
   - Differentiate keys by file 체크 해제 (안 하면 데이터 오염)
   - Replace modified values 선택
   - Detected language 확인
7. 업로드 완료 알림 확인
8. Sync 페이지 접속 → Email / Branch / Mode 입력
9. "Just Do It" 클릭 → 10분 대기
10. 완료 팝업 확인 → Fork에 자동 푸시됨 확인
```

**1회 소요 시간: 약 30~60분 (오류 발생 시 더 길어짐)**

### 반복 빈도

- 스트링 수정이 발생할 때마다 전 과정 반복
- 런칭/업데이트 시즌: 주 수회 이상 반복
- 긴급 수정(핫픽스): 퇴근 후/주말에도 대응 필요

### 구조적 고충

| 고충 | 내용 |
|---|---|
| **본업 방해** | PM 업무 중 갑자기 번역 요청이 들어오면 하던 일 중단 |
| **단일 장애점** | 담당자 부재(휴가, 병가) 시 번역 파이프라인 전체 중단 |
| **실수 위험** | 설정값 수동 입력 → `Differentiate keys by file` 실수 시 전체 데이터 오염 |
| **충돌 위험** | Sync Lambda push 타이밍이 다른 작업자 작업과 겹치면 git 충돌 |
| **확인 부담** | 10분 대기 후 결과 확인을 직접 해야 함 (알림 없음) |
| **의미 없는 전문성** | 번역 내용을 이해하지 못해도 수행 가능한 단순 반복 작업 |

## 2. 현재 시스템 전체 구조

### 데이터 흐름

```
[기획팀] excel/*.xlsx에 한글 텍스트 직접 입력
  예: cookies.xlsx → name = "용감한 쿠키"
         ↓
[datasheet 도구] 한글 텍스트 추출 → translations.xlsx 자동 생성
  예: Cookies.cookies^110001.name → "용감한 쿠키"
         ↓
[PM - 수동] "번역키 추출" 버튼 → exports/ 파일 생성
         ↓
[PM - 수동] 파일 7종 카테고리 분류
         ↓
[PM - 수동] lokalise 업로드 (설정값 수동)
         ↓
[번역팀] lokalise에서 7개 언어 번역 입력
         ↓
[PM - 수동] Sync Lambda → cos-data에 push
         ↓
[datasheet] translations_ko/en/ja/zh-TW/zh-CN/th/id.pb 생성
         ↓
[배포] 게임 서버/클라이언트에 번역 반영
```

### 현재 파일 현황

| 파일 | 역할 | 항목 수 |
|---|---|---|
| `translations/translations.xlsx` | 한국어 스트링 중앙 DB | **8,546개** |
| `translations/changes/*.xlsx` | 변경 이력 (타임스탬프 파일) | 50개 (최근 9일) |
| `excel/ui_strings.xlsx` | UI 전용 스트링 (key 방식) | 별도 관리 |
| `excel/*.xlsx` (131개) | 게임 데이터 (한글 직접 입력) | - |

### 스트링 key 구조

```
현재 자동 생성 key 형식:
  TableName.sheetName^ID.columnName
  예) Cookies.cookies^110001.name
      AttendanceEvents.events^1.description
      BountyModes.bounty_modes^1.strings^0

ui_strings 전용 key 형식 (이미 key 방식):
  Pre_[Screen]_[Component]_[Action]
  예) Pre_UI_Common_Button_Confirm
```

## 3. 문제점 진단

### 구조적 문제

| 문제 | 원인 | 영향 |
|---|---|---|
| **PM이 인간 번역기 역할** | 번역팀↔cos-data 사이에 자동 연결 없음 | 담당자 번아웃, 단일 장애점 |
| **lokalise 중간 플랫폼 의존** | 번역 관리 전용 플랫폼 필요 | 구독 비용 + 별도 계정 관리 |
| **수동 파일 분류** | 추출 후 카테고리 분류 도구 없음 | 매번 30~60분 소요 |
| **Sync 타이밍 충돌** | 수동 push → 다른 작업자와 충돌 가능 | git 충돌 → Revert 발생 |
| **번역 현황 파악 불가** | 어떤 스트링이 번역됐는지 한눈에 안 보임 | 누락 스트링 발견 늦음 |

### 번역팀 접근성 문제

- lokalise 계정이 한정적 → 계정 권한 요청 → PM 경유 → 지연
- 번역팀이 직접 현황을 볼 수 없음 → PM이 중간에서 커뮤니케이션 담당
- 어떤 스트링이 new/수정/완료인지 태깅 체계 없음

## 4. 제안: Google Sheet 기반 통합 번역 파이프라인

### 핵심 개념

> **Google Sheet = 모든 언어 스트링의 단일 소스 오브 트루스**
> 기획자는 key + ko만 입력. 번역팀은 전용 툴로 나머지 언어 입력.
> PM의 중간 역할 100% 제거.

### 기획자 작업 방식

기획자가 하는 일은 2가지뿐:

```
1. 게임 데이터 시트 (excel)
   |char_id | char_name        |
   |1000    | str_cookie_power |  ← 값 대신 key 참조

2. 스트링 시트 (Google Sheet — 직접 입력)
   |str_id            | tag | ko      | en | ja | ... |
   |str_cookie_power  | new | 힘쎈쿠키 |    |    |     |  ← ko + tag만 입력
```

en/ja 등 나머지 언어는 **번역팀이 채운다.** 기획자는 신경 쓰지 않아도 됨.

### Google Sheet 구조

```
| str_id           | tag  | ko       | en           | ja              | zh-TW | zh-CN | th | id |
|------------------|------|----------|--------------|-----------------|-------|-------|----|----|
| str_cookie_power | done | 힘쎈쿠키  | Brave Cookie | ブレイブクッキー  | ...   | ...   | ...| ...|
| str_new_event    | done | 신규이벤트 | New Event    | ...             | ...   | ...   | ...| ...|
| str_skill_blast  | new  | 블라스트  |              |                 |       |       |    |    |
| str_item_shield  | re   | 보호막(수정)| Shield      | ...             | ...   | ...   | ...| ...|
```

**tag 값 정의**:

| tag | 의미 |
|---|---|
| `new` | 신규 스트링 — 번역 필요 |
| `re` | ko 수정됨 — 재번역 필요 |
| `done` | 번역 완료 |
| `hold` | 기획 확정 전 보류 |

### 번역팀 전용 워크벤치 툴

Google Sheet 전체(수천 개)를 번역팀이 직접 보기 어려우므로 별도 툴 사용:

```
[번역팀 워크벤치 툴]
Google Sheet에서 tag=new/re 항목만 로드
    ↓
| str_id           | ko      | en | ja | ... |
| str_skill_blast  | 블라스트 |    |    |     |  ← 입력
| str_item_shield  | 보호막   |    |    |     |  ← 입력

[저장] 클릭
    ↓
Google Sheet 해당 셀 업데이트 + tag = done 자동 변경
```

구현 난이도 낮음 — Google Sheets API read/write + 단순 테이블 UI.

### 배포 모드 2가지

#### 모드 1 — ko CSV 자동 배포 (Unity Editor 개발용)

```
[CI cron — 5분마다]
Google Sheet ko 컬럼 전체 읽기
    ↓
이전 hash와 비교
    ├── 동일 → skip (API 호출만, 실질 부하 없음)
    └── 다름 → ko_csv 재생성 → cos-client에 자동 배포
                                        ↓
                             Unity Editor가 CSV 읽어 즉시 반영
```

- 기획자가 Google Sheet에 ko 입력 후 **5분 내 Unity Editor에 자동 반영**
- 번역 없이 ko만으로 개발/확인 가능
- 폰 빌드는 어차피 새로 만들어야 하므로 이 CSV로 충분

#### 모드 2 — 전체 언어 자동 배포 (1일 1회) + 긴급 수동 배포

**기본: 매일 새벽 자동 배포**

```
[CI cron — 매일 새벽 4시]
Google Sheet 전체 읽기 (ko + en + ja + zh-TW + zh-CN + th + id)
    → 전체 언어 pb 생성 → Portal 업로드
    → Unity에서 전체 언어 확인 가능 / 폰 빌드에 반영
    → Slack 배포 완료 알림
```

**긴급: 기획 팀장급 수동 배포**

```
[기획 팀장] 별도 배포 툴에서 "전체 언어 배포" 실행
    ↓
CI 즉시 트리거 → 전체 언어 pb 생성 → Portal 업로드
```

- 번역팀은 번역만 담당 — 배포 권한/책임 없음
- 핫픽스·긴급 업데이트 시 팀장급이 판단 후 수동 실행
- 배포 툴: 간단한 웹 UI 또는 Slack slash command 형태

## 8. 번역 데이터에 커밋별 히스토리가 불필요한 이유

### 코드 vs 번역 데이터의 본질적 차이

| 항목 | 코드 / 게임 데이터 | 번역 데이터 |
|---|---|---|
| 과거 버전의 의미 | 버그 추적, 롤백 기준점 | "예전에 틀렸던 번역" → 쓸모 없음 |
| 최신 = 정답 | 아닐 수 있음 (버그 도입 가능) | **항상 맞음** |
| 롤백 시나리오 | 빈번함 | 거의 없음 (틀리면 고치면 끝) |
| diff의 유용성 | 변경 원인 추적에 필수 | pb 바이너리 → git diff 자체가 불가 |
| 히스토리 필요 | ✅ 필수 | ❌ 불필요 |

### 번역 히스토리가 필요 없는 구체적 이유

**1. "최신이 항상 정답"인 데이터 특성**
- 번역을 수정하면 → 수정한 것이 맞는 것
- 이전 번역으로 롤백해야 할 이유가 없음
- 코드와 달리 "이전 번역이 더 좋았다"는 판단은 번역팀이 직접 재수정하면 됨

**2. git diff가 번역에 의미 없음**
- 번역 결과물은 pb 바이너리 파일 → `git diff`로 내용 확인 불가
- 현재도 번역 변경 추적을 위해 `translations/changes/*.xlsx`를 별도로 50개 쌓고 있음
- 즉, git 히스토리가 실질적 역할을 못 해서 별도 시스템을 만든 것

**3. 의미 있는 추적 단위는 "커밋"이 아니라 "배포"**
- 번역 관련 질문은 항상 "v2.3에 뭐가 나갔지?"이지, "3월 27일 14:32 커밋에 뭐가 바뀌었지?"가 아님
- 배포 시점 스냅샷이 훨씬 실용적인 추적 단위

**4. 저장 비용 대비 효용이 없음**
- 현재 cos-data `.git` 4.6GB — 번역 pb 누적이 주요 원인 중 하나
- 매 번역 수정마다 pb 전체가 binary로 커밋 → 용량 지속 증가
- 히스토리가 의미 없는데 용량만 차지하는 구조

### 배포 시점 스냅샷으로 충분한 이유

| 필요한 상황 | 스냅샷으로 해결 가능한가 |
|---|---|
| "v2.3 라이브 당시 번역 확인" | ✅ `translations_snapshot_v2.3.0_*.xlsx` 열면 끝 |
| "v2.3 → v2.4 사이 변경 내용" | ✅ 두 스냅샷 비교 (xlsx이라 사람이 읽기도 쉬움) |
| "라이브 번역 오류 클레임 대응" | ✅ 해당 버전 스냅샷으로 원문 확인 |
| "누가 언제 번역했는지" | ✅ Google Sheet 리비전 히스토리로 충분 |
| "특정 커밋 시점 번역 복원" | ❌ 불필요 — 번역팀이 재수정하면 됨 |

### 스냅샷 저장 구조

```
S3 (기존 업로드 인프라 활용)
  └── translations-snapshots/
        ├── translations_snapshot_v2.3.0_2026-03-15.xlsx
        ├── translations_snapshot_v2.4.0_2026-04-01.xlsx
        └── translations_snapshot_v2.4.1_2026-04-04.xlsx  ← 핫픽스도 포함
```

- 배포 트리거 시 자동 생성 → S3 업로드
- Slack 배포 완료 알림에 스냅샷 링크 포함
- xlsx 포맷 → 별도 도구 없이 바로 열어볼 수 있음

## 8-2. 실행 로드맵

### 기본 원칙: 병행 개발 → 일괄 cut-over

현재 구조(excel 직접 입력 + lokalise)와 새로운 구조(str_id 참조 + Google Sheet)는 **중간 상태가 없다.**

## 9. 기존 vs 새로운 방식 비교

### 작업 흐름 비교

| 단계 | 기존 방식 | 새로운 방식 |
|---|---|---|
| 스트링 추가 | 기획팀이 excel에 한글 입력 | 기획팀이 excel에 한글 입력 (동일) |
| 추출 | PM이 Fork → 버튼 클릭 → 20초 대기 | datasheet 실행 시 자동 감지 |
| 분류 | PM이 7개 카테고리로 수동 분류 | **없음** |
| 번역 플랫폼 등록 | PM이 lokalise에 파일별 업로드 | **없음** |
| 번역팀 알림 | PM이 별도로 슬랙/메일 연락 | Slack 자동 알림 |

| 번역 현황 확인 | PM이 lokalise 로그인해서 확인 | Google Sheet에서 실시간 확인 |

## 11. 장기 방향 (별도 검토)

현재 `excel/*.xlsx`에는 한글 텍스트가 직접 입력되어 있고, datasheet가 이를 자동 추출해서 key를 생성하는 구조. 이 구조는 당장 유지하되, 장기적으로는:

- **모든 excel 텍스트 컬럼 → key 참조 방식 전환**
- `cookies.xlsx`: `name = 용감한 쿠키` → `name_key = Cookies.cookies^110001.name`
- datasheet 도구 수정 필요 (cos-data-manager 팀)
- 전환 시 Claude를 활용한 131개 파일 자동 마이그레이션 가능

## 별첨 — VNG 외부 법인 번역 파이프라인 검토

### 배경

VNG는 완전 외부 법인으로, 내부 번역팀과 다른 접근 방식이 필요하다.

| 항목 | 내부 번역팀 | VNG |
|---|---|---|
| 관계 | 같은 회사, 다른 조직 | 완전 외부 법인 |
| Google Sheet 직접 접근 | ✅ 가능 | ❌ 보안/IP 이슈 |
| 번역 언어 | en/ja/zh-TW/zh-CN/th/id | vi (베트남어) + 해당 언어 |
| 배포 인프라 | Portal | VNG 별도 AWS (ap-southeast-5) |

---

### 옵션 A — Export/Import 파일 교환 방식

#### 흐름

```
[워크벤치 툴]
  "VNG 번역 요청 내보내기" 버튼
      ↓
  tag=new/re 중 VNG 대상 항목만 추출 → xlsx/csv 파일 생성
      ↓
  COS팀 → VNG에게 파일 전달 (이메일 / 공유 폴더)
      ↓
  [VNG] 번역 완료 후 파일 반환
      ↓
  "VNG 번역 파일 가져오기" 버튼
      ↓
  메인 Google Sheet에 자동 머지 + tag=done
```

#### 장단점

| 구분 | 내용 |
|---|---|
| **장점** | VNG에게 시스템 접근 권한 불필요 — 보안 이슈 없음 |
| **장점** | 파일 교환 방식이라 VNG 측 적응 쉬움 |
| **장점** | 워크벤치 툴에 Export/Import 기능만 추가하면 됨 — 구현 단순 |
| **장점** | VNG 번역 파일을 COS팀이 검수 후 반영 가능 |
| **단점** | 파일 전달·반환 과정에 사람이 개입 (완전 자동화 불가) |
| **단점** | 파일 버전 혼선 가능 (어떤 파일이 최신인지) |
| **단점** | 번역 완료 시점을 COS팀이 직접 확인해야 함 |

#### 구현 난이도

- 워크벤치 툴에 xlsx/csv Export + Import 기능 추가
- 난이도: **낮음** (기존 툴 기능 확장)

### 옵션 B — VNG 전용 별도 Google Sheet 연동

#### 흐름

```
메인 Google Sheet (내부 — COS팀 관리)
      ↓ CI 자동 동기화 (tag=new/re 항목만)
VNG 전용 Google Sheet (외부 공유 — VNG 접근 가능)
      ↓
[VNG] 번역 입력 → vi 컬럼 채움
      ↓ CI 자동 감지 (5분 cron 또는 webhook)
메인 Google Sheet에 vi 컬럼 자동 머지 + tag=done
```

#### VNG 전용 시트 구조

```
| str_id           | ko      | vi          | tag  |
|------------------|---------|-------------|------|
| str_cookie_power | 힘쎈쿠키 |             | new  |  ← VNG가 vi 입력
| str_new_event    | 신규이벤트|             | new  |
```

#### 장단점

| 구분 | 내용 |
|---|---|
| **장점** | VNG가 자체 속도로 작업 가능 — COS팀 개입 없음 |
| **장점** | 번역 현황 실시간 확인 가능 (시트에서 직접) |
| **장점** | 파일 전달/반환 과정 없음 — 더 빠른 번역 반영 |
| **단점** | VNG 전용 시트 ↔ 메인 시트 동기화 로직 추가 개발 필요 |
| **단점** | Google Sheet 외부 공유 정책 확인 필요 |
| **단점** | 동기화 오류 시 데이터 불일치 리스크 |
| **단점** | VNG 인원 변경 시 시트 권한 관리 필요 |

#### 구현 난이도

- VNG 전용 시트 → 메인 시트 머지 CI 추가
- 난이도: **중간**

### 비교 요약

| 항목 | 옵션 A (파일 교환) | 옵션 B (별도 시트) |
|---|---|---|

### 배포 파이프라인

```
전체 언어 pb 새벽 4시 cron

```
